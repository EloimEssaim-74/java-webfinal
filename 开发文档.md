## 1. 项目概述

### 1.1 项目背景
在内容为王的时代，兼具社区互动与智能创作辅助的知识平台拥有巨大的商业与技术价值。本项目旨在构建一个“具有AI能力的知乎后台”系统，在用户、文章、互动等基础功能之上，深入攻克高并发网关限流、实时热榜、异步广播处理及AI流式写作辅助等核心技术挑战。

### 1.2 项目目标
- 构建完整的内容发布与互动平台后端服务。
- 落地微服务架构，实现动态限流、热搜榜单、消息异步广播、AI流式输出等高级特性。
- 引入多级缓存、读写分离、异步削峰等机制，使平台具备支撑数万QPS的高并发能力。
- 通过Docker Compose实现一键部署，保障环境一致性与快速交付。

### 1.3 范围定义
平台功能分为两部分：
- **第一部分：基础支撑功能**
  用户权限管理，文章创建、发布、修改与逻辑删除，评论与点赞。
- **第二部分：核心架构功能**
  基于配置中心的动态网关限流，Redis ZSET实时热搜Top10，RabbitMQ广播处理长文，SSE流式AI续写，以及高并发扩展设计。

## 2. 需求分析

### 2.1 功能需求

#### 2.1.1 基础支撑功能
- **用户权限**：注册、登录、JWT Token签发与校验、注销，区分普通用户与管理员角色。
- **文章管理**：创建（草稿态）、发布、修改、逻辑删除；分页获取最新列表；获取文章详情。
- **基础互动**：评论入库；单次点赞更新（每人每文章仅一次点赞，简化处理为每次调用更新计数）。

#### 2.1.2 核心架构功能
- **动态网关限流**：网关层对IP或接口维度限流，参数由Nacos配置中心动态下发，无需重启。
- **实时热搜榜**：基于Redis ZSET记录文章热度，提供全站Top 10热榜API。
- **长文异步广播**：文章发布后投递至RabbitMQ Fanout交换机，多个下游消费者（AI标签提取、合规检测等）同时独立处理。
- **AI流式创作辅助**：提供续写API，采用SSE流式响应逐句推送AI生成文本。

### 2.2 非功能需求
- **高并发**：核心读接口支撑10k+ QPS，写接口支撑3k+ QPS。
- **高可用**：服务多实例部署，数据库主从复制，缓存冗余。
- **可扩展性**：服务按功能拆分，支持无状态水平扩展。
- **安全性**：JWT校验、角色权限控制、限流防爬。
- **可维护性**：配置集中管理，容器化部署，统一日志收集。

## 4. 详细功能设计

### 4.1 用户权限模块 (user-service)
- **注册**：接收用户名、密码（BCrypt加密）、角色（默认`user`），存入`users`表。
- **登录**：校验密码，生成含用户ID与角色的JWT，返回Token。
- **JWT校验**：网关过滤器解析`Authorization`头，验证签名与有效期，并检查Redis黑名单；通过后将用户信息设置到请求头传递给下游。
- **注销**：将Token存入Redis黑名单，有效期与原Token一致。
- **角色权限**：普通用户操作自身资源；管理员可访问`/admin/*`接口（网关根据角色拦截）。

### 4.2 文章内容管理 (article-service)
- **创建文章**：接收标题、内容、状态（`DRAFT`/`PUBLISHED`），返回文章ID。
- **发布文章**：状态置为`PUBLISHED`，并发送消息至RabbitMQ（`article.publish`事件）。
- **修改文章**：仅作者或管理员可修改。
- **逻辑删除**：设置`deleted=1`，不物理删除。
- **分页最新列表**：按`create_time`降序，支持分页参数。
- **文章详情**：返回文章内容，同时触发阅读热度增加（调用Redis `ZINCRBY hot_articles`，并结合防重逻辑）。

### 4.3 互动基础功能 (interact-service)
- **评论入库**：接收文章ID、用户ID、评论内容，写入Redis事件队列（`comment:events`），由后台消费者批量持久化至MySQL，并触发热度微增。
- **单次点赞**：
  1. 利用Redis键`like:article:{articleId}:user:{userId}`防重。
  2. 点赞事件写入Redis Stream `like:events`，立即返回成功。
  3. 消费者`like-persist-consumer`批量取出事件，进行幂等校验后更新MySQL文章点赞计数，并执行`ZINCRBY`更新热度。

### 4.4 动态网关限流 (gateway-service + Nacos)
- 使用 **Sentinel** 集成 Spring Cloud Gateway。
- 限流规则存储在 Nacos 的 `sentinel-gateway-rules` 配置中，格式示例：
```json
[
  {
    "resource": "article_detail_route",
    "grade": 1,
    "count": 5,
    "intervalSec": 1,
    "controlBehavior": 0,
    "paramItem": {
      "parseStrategy": 3,
      "fieldName": "X-Real-IP"
    }
  }
]
```
- 规则变更后，通过 Nacos 动态监听与 `@RefreshScope` 实时生效，或由 Sentinel Dashboard 推送至 Nacos。
- 多实例部署时，采用 **Sentinel 集群模式**（Token Server）实现全局限流。
- 触发限流时返回 `429 Too Many Requests` 及提示信息。

### 4.5 文章实时热搜榜
- **数据结构**：Redis Sorted Set `hot_articles`，member 为文章ID，score 为热度分。
- **热度分规则**：阅读 +1，点赞 +3。同一用户短时间重复阅读仅计一次（`read:article:{id}:user:{uid}` TTL 5分钟）。
- **Top 10 查询**：`ZREVRANGE hot_articles 0 9 WITHSCORES`，然后批量从缓存/MySQL补齐标题等信息返回。
- **防霸榜**：每日定时任务将历史分数乘以衰减因子（如0.9）。
- **高性能优化**：Top 10 结果用 Caffeine 本地缓存 TTL 1秒，通过 Redis Pub/Sub 热度变更事件主动刷新。

### 4.6 长文异步广播处理 (article-service + RabbitMQ)
- **消息格式**：包含文章ID、标题、内容、发布时间。
- **交换机**：采用 `article.topic.exchange`，路由键分别为 `article.tag` 和 `article.compliance`，两个独立队列消费，避免处理速度差异互相影响。
- **tag-extract-service**：消费消息，调用AI提取关键词/标签，回写文章表的`tags`字段。
- **compliance-service**：消费消息，调用AI内容审核接口，若违规则标记或自动逻辑删除。
- 两个消费者均配置并发线程（如5-10），消息持久化并手动ACK，保证可靠性。

### 4.7 AI流式创作辅助 (ai-assistant-service)
- **API**：`POST /api/ai/continue`，参数包含上下文、用户ID。
- **实现**：基于 Spring WebFlux，返回 `MediaType.TEXT_EVENT_STREAM` 的 `Flux<ServerSentEvent<String>>`。
- **处理流程**：
  1. 构建 Prompt，调用 OpenAI 兼容的流式 Chat Completion API。
  2. 将响应中的 `data` 块映射为 SSE 事件，通过 Reactor 流推送至客户端。
  3. 设置超时、心跳保活，结合 Resilience4j CircuitBreaker 熔断，超限时返回降级文案。
- 使用 Reactor Netty 连接池管理 LLM 后端连接，Sentinel 限制并发连接数。

## 6. API接口设计（核心部分）

### 6.1 用户服务
- `POST /api/user/register` – 注册
- `POST /api/user/login` – 登录，返回JWT
- `POST /api/user/logout` – 注销（Header带Token）

- `POST /api/articles` – 新建（草稿）
- `PUT /api/articles/{id}/publish` – 发布
- `PUT /api/articles/{id}` – 修改
- `DELETE /api/articles/{id}` – 逻辑删除
- `GET /api/articles?page=1&size=20` – 最新列表
- `GET /api/articles/{id}` – 详情（触发阅读热度）

### 6.3 互动服务
- `POST /api/comments` – 添加评论
- `POST /api/articles/{id}/like` – 点赞

### 6.4 热搜服务
- `GET /api/trending` – 获取全站热搜Top 10

### 6.5 AI服务
- `POST /api/ai/continue` – 流式续写（SSE）

## 7. 配置中心与动态限流方案

- **Gateway监听**：
```java
@RefreshScope
@Configuration
public class GatewayConfig {
    @Value("${sentinel.rules:}")
    private String sentinelRules;
    // 在Bean初始化时解析JSON并加载到Sentinel规则管理器

```
- 修改Nacos配置后，网关自动刷新，实现动态限流。

## 8. Docker部署方案

### 8.1 容器化策略
- 微服务基于 `openjdk:17-jdk-alpine` 打包成独立镜像。
- 基础中间件采用官方镜像：MySQL 8.0、Redis 7、RabbitMQ 3-management、Nacos 2.2、Nginx。
- 使用Docker Compose编排，网络统一为 `kb-network`。
- 数据持久化通过 volumes 映射，支持 `docker-compose up --scale` 扩展服务实例。

      - DB_HOST=mysql-master
    networks:
      - kb-network

volumes:
  mysql_master_data:
  mysql_slave_data:
  redis_data:
  rabbitmq_data:
  nacos_data:

networks:
  kb-network:

### 8.5 部署步骤
1. 安装 Docker 及 Docker Compose。
2. 复制配置文件（nginx.conf、SQL初始化脚本等）至 `config/` 目录。
3. 各微服务目录执行 `mvn clean package` 生成 jar。
4. 在项目根目录执行 `docker-compose up -d`。
5. 通过 `docker-compose logs -f` 检查启动状态。
6. 访问 `http://localhost` 测试接口，Nacos 控制台 `http://localhost:8848/nacos`。
7. 扩展服务实例：`docker-compose up --scale article-service=3 -d`。

## 10. 测试策略

- **单元测试**：JUnit 5 + Mockito，覆盖核心 Service 逻辑。
- **接口测试**：Spring MockMvc 或 Postman 集合，覆盖正向、异常、鉴权场景。
- **限流测试**：JMeter 高并发请求文章详情，验证 Sentinel 限流触发。
- **流式测试**：curl 请求 SSE 接口，检查逐块输出与超时断连处理。
- **消息队列测试**：发布文章后检查消费者日志，确认标签与合规结果写入。
- **高并发压测**：JMeter 混合场景模拟，逐步加压至目标峰值，监控系统瓶颈。
- **集成测试**：全服务启动，通过 Nginx 网关访问，端到端验证。

- **日志收集**：服务日志输出为 JSON 格式，可接入 ELK 或 Loki。
