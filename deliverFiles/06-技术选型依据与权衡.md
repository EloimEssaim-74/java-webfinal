# 06-技术选型依据与权衡

> **文档版本**: 1.0  
> **最后更新**: 2026-07-06  
> **适用分支**: stage6+  
> **适用范围**: Knowledge Platform 全栈  

---

## 1. 概述

本文档记录了 Knowledge Platform（智能知识库与内容发布平台）项目中每一项核心技术选型的决策依据。每项决策均包含：为什么选择该技术、权衡了什么、以及未来什么条件下需要重新评估。

### 1.1 评估维度

每项技术选型按照以下维度进行评估：

| 维度 | 权重 | 说明 |
|------|------|------|
| 团队熟悉度 | 高 | 当前阶段最关键因素，降低学习曲线和维护成本 |
| 生态成熟度 | 高 | 文档丰富度、社区活跃度、问题可搜索性 |
| 性能 | 中 | 当前规模（< 1000 QPS）下非瓶颈 |
| 运维复杂度 | 中 | Docker Compose 单机部署，运维成本需可控 |
| 扩展性 | 中 | 预留向 K8s 迁移的空间 |
| 许可证/成本 | 低 | 全部开源，无商业许可问题 |

## 2. 后端核心框架

### 2.1 基础框架：Spring Boot 3.2.6 + Spring Cloud 2023.0.3

| 项目 | 内容 |
|------|------|
| **选择** | Spring Boot 3.2.6 + Spring Cloud 2023.0.3 |
| **版本来源** | `pom.xml` — `spring-boot-starter-parent:3.2.6`, `spring-cloud.version:2023.0.3` |
| **Java 版本** | Java 17 (LTS) |

**为什么选择**：

1. **生态成熟度**：Spring 是 Java 企业开发的事实标准。从 DI/IoC、AOP、事务管理到 Spring Cloud 微服务全家桶，形成了一套经过大规模生产验证的完整解决方案。遇到问题时，Stack Overflow、Spring 官方文档、社区博客的覆盖密度远超其他框架。

2. **团队熟悉度**：团队成员具备 Spring Boot 开发经验，无需额外的学习曲线即可进入开发。在当前项目周期（数周）内，团队熟悉度是最关键的效率因子。

3. **Spring Cloud 集成**：Spring Cloud Gateway（网关）、Spring Cloud LoadBalancer（客户端负载均衡）、Spring Cloud CircuitBreaker（Resilience4j 熔断）三者无缝集成，配置一致性高，减少了异构组件集成的胶水代码。

**权衡**：

- **启动速度**：Spring Boot 应用的启动时间（5-15 秒）慢于 Quarkus（亚秒级）。但在 Docker Compose 单机部署场景下，服务重启频率低，启动时间不构成瓶颈。
- **内存占用**：每个微服务 JVM 基础内存约 150-250MB，高于 Go/Rust 服务（10-50MB）。当前 8 个服务合计约 1.5-2GB，仍在单机可承受范围内。
- **GraalVM 原生镜像**：Spring Boot 3.x 支持 AOT 编译为原生镜像，可显著降低内存和启动时间，但需要额外配置（反射、动态代理声明），当前阶段未启用。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Quarkus** | 亚秒级启动，低内存，原生 K8s 支持 | 生态规模约为 Spring 的 1/10，部分 Spring 生态组件需要适配 | 团队不熟悉，排查问题时可用资源少 |
| **Micronaut** | 编译时 DI，低内存，快速启动 | 社区更小，与 Nacos/Sentinel 等阿里系组件无官方集成 | 集成风险高 |
| **Go + Gin** | 极低内存，编译为单二进制 | 需要团队学习新语言，ORM/事务/安全等需重新选型 | 语言切换成本过高 |

**什么时候重新评估**：

- 迁移至 K8s + Serverless 架构时（冷启动时间成为瓶颈），评估 GraalVM 原生镜像编译或 Quarkus
- 团队规模扩大、有 Rust/Go 人力储备时，可考虑将非核心计算型服务（如 tag-extract-service）用 Go 重写

### 2.2 API 网关：Spring Cloud Gateway

| 项目 | 内容 |
|------|------|
| **选择** | Spring Cloud Gateway (基于 WebFlux/Reactor Netty) |
| **版本** | 随 Spring Cloud 2023.0.3 |
| **实例数** | 2（`gateway-1`, `gateway-2`，least_conn 负载均衡） |

**为什么选择**：

1. **Reactive 非阻塞**：Gateway 作为所有请求的入口，必须是非阻塞架构才能应对高并发。Spring Cloud Gateway 基于 WebFlux + Reactor Netty，单实例可轻松处理数万并发连接，不会像 Tomcat 线程池模型那样在线程上下文切换上消耗大量资源。

2. **Spring 生态无缝集成**：与 Nacos 服务发现、Sentinel 限流、Resilience4j 熔断均通过 Spring Cloud 抽象层集成，无需编写适配器代码。例如，`lb://article-service` 自动通过 Nacos 发现服务实例并负载均衡。

3. **内置 Sentinel 支持**：Sentinel SCG Adapter 提供 `GatewayFlowRule` 原生支持，可基于 route ID、路径模式、请求参数等维度限流，无需自建过滤器。

**权衡**：

- **内存占用**：每个 Gateway 实例约 256-512MB 堆内存，两个实例合计约 1GB。对于纯转发网关而言偏高，但仍在可接受范围。
- **WebFlux 调试**：响应式编程的堆栈追踪不如 Servlet 直观，调试时需要熟悉 Reactor 的 `Hooks.onOperatorDebug()`。
- **功能范围**：Gateway 仅支持 API 路由/限流/认证，不支持 TCP 代理或 gRPC 代理。当前所有服务间通信走 HTTP，暂不需要。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Kong** | 高性能，插件市场丰富，支持 TCP/gRPC | 需要独立部署和管理 PostgreSQL/Cassandra，运维复杂度显著增加 | 当前规模不需要 Kong 的插件生态，运维成本大于收益 |
| **Zuul 1.x** | Netlix 出品，Servlet 同步模型易理解 | 阻塞 I/O，性能瓶颈，已停止维护 | 技术债务风险 |
| **Traefik** | 自动服务发现，Let's Encrypt 集成 | 与 Nacos 无原生集成，需额外配置 | 集成成本高 |
| **Nginx + Lua (OpenResty)** | 极致性能，C 级别 | 复杂逻辑需 Lua 脚本，维护成本高 | 网关逻辑（JWT 验证、Sentinel 集成）复杂，Lua 开发效率低 |

**什么时候重新评估**：

- 当需要 TCP/gRPC 代理时，评估 Kong 或 Traefik
- 当网关成为性能瓶颈（> 10K QPS 单实例），评估 OpenResty 或 Envoy
- 迁移 K8s 后，评估是否用 Ingress Controller + Gateway API 替代

### 2.3 服务注册与配置：Nacos 2.2

| 项目 | 内容 |
|------|------|
| **选择** | Nacos 2.2.0 (standalone mode) |
| **依赖** | `spring-cloud-alibaba.version:2023.0.1.2` |

**为什么选择**：

1. **注册中心 + 配置中心合一**：无需额外部署 Spring Cloud Config Server 或 Apollo。一套 Nacos 同时管理服务注册发现和动态配置（Sentinel 限流规则、功能开关等），运维成本减半。

2. **gRPC 推送**：Nacos 2.x 使用 gRPC 长连接推送配置变更，替代 1.x 的 HTTP 轮询。Sentinel 规则变更可在秒级内推送到所有 Gateway 实例，无需重启。

3. **阿里生态兼容**：与 Sentinel、RocketMQ（未来备选）等同属阿里中间件生态，版本兼容性有保障。

**权衡**：

- **单点风险**：当前使用 standalone 模式部署，Nacos 宕机则新服务无法注册、配置无法更新。已在 `pom.xml` 中引入 `spring-cloud-starter-loadbalancer` 作为客户端侧兜底（已注册服务可继续通过本地缓存调用）。
- **资源占用**：Nacos 2.x Java 进程内存约 512MB-1GB，比 Eureka（~200MB）重。但合一了配置中心，总资源消耗低于 Eureka + Config Server 组合。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Eureka + Config Server** | Spring 官方，轻量 | 配置与注册分离，需部署两套 | 运维成本 2x |
| **Consul** | 健康检查强，多数据中心支持 | 比 Nacos 重，配置管理 UI 不如 Nacos | 运维复杂度更高 |
| **ZooKeeper** | CP 强一致性 | 不适合高并发读写（基于 Paxos），配置管理需自建 | 老旧，不推荐新项目 |

**什么时候重新评估**：

- 当前 Nacos standalone 在单机部署下足够。迁移至多机房/多活架构时，升级为 Nacos 集群模式（3 节点 Raft）。

## 3. 流量控制与限流

### 3.1 双层限流：Sentinel + Nginx

| 层级 | 技术 | 维度 | 配置来源 |
|------|------|------|---------|
| L1 (边缘) | Nginx `limit_req_zone` | 单 IP | nginx.conf 静态配置 |
| L2 (网关) | Sentinel Gateway Flow Rule | 路由/API 粒度 | Nacos 动态推送 |

**为什么选择双层架构**：

1. **纵深防御**：L1 Nginx 层拒绝大部分恶意流量（如单 IP 高频刷接口），减轻 Gateway 压力。L2 Sentinel 层提供更细粒度的控制（如对 `/api/user/login` 单独限流 10 QPS），并可动态调整。

2. **Sentinel 动态规则**：Sentinel 规则通过 Nacos 推送，运维人员可在运行时调整限流阈值，无需重启 Gateway：

   ```java
   // 文件: gateway-service/src/main/java/com/kb/gateway/config/SentinelRuleBootstrap.java
   // 从 Nacos 读取 gateway-sentinel-rules，注册 gRPC listener
   // 规则变更实时生效，无重启
   ```

3. **Nginx 高性能拒绝**：Nginx 的令牌桶算法在 C 层面执行，拒绝一个请求的 CPU 开销远低于让请求进入 Java Gateway 后再被 Sentinel 拒绝。

**权衡**：

- **两套限流配置**：Nginx 静态限流 + Sentinel 动态限流需要分别在两处配置，增加了理解和维护成本。但分层职责清晰，长期来看利大于弊。
- **Sentinel Dashboard**：Sentinel 控制台（端口 18080）在开发环境使用，生产环境当前未启用持久化。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **纯 Nginx 限流** | 极高性能 | 不支持动态修改，无精细 API 级别控制 | 无法满足动态运营需求 |
| **纯 Sentinel 限流** | 动态灵活，API 粒度 | 恶意流量先穿透 Nginx 到达 Gateway，多消耗一层资源 | 缺少边缘防御 |
| **Redis 令牌桶** | 多实例共享限流计数 | 每次限流判定需要一次 Redis 网络调用，增加 ~1ms 延迟 | 性能开销过高 |
| **Guava RateLimiter** | 单机内存级，极低延迟 | 不支持分布式（多 Gateway 实例各自计数），不支持动态变更 | 无法在多实例场景下有效限流 |

**什么时候重新评估**：

- QPS > 10K 时，Sentinel 本身可能成为瓶颈，评估将限流逻辑下沉至 Nginx + Redis（使用 `redis-rate-limiter` 模块）
- 迁移 K8s 后，评估使用 Istio/Envoy 的 RateLimit 功能替代部分 Sentinel 职责

## 4. 数据持久化

### 4.1 ORM：MyBatis-Plus 3.5.7

| 项目 | 内容 |
|------|------|
| **选择** | MyBatis-Plus 3.5.7 + `mybatis-plus-spring-boot3-starter` |
| **版本** | `3.5.7` |

**为什么选择**：

1. **SQL 可控**：复杂查询（如热搜 Top 10 带条件过滤、文章列表多条件检索）可以通过手写 SQL 精确控制执行计划和索引使用，避免 ORM 自动生成的 N+1 查询或低效 JOIN。在 MyBatis XML mapper 中可以清晰地看到每一次数据库交互。

2. **开箱即用的功能**：逻辑删除（`deleted` 字段自动处理）、自动填充（创建时间/更新时间）、分页插件（物理分页）、乐观锁——这些企业应用刚需功能只需注解配置，无需手写。

3. **与 JPA 的对比**：JPA/Hibernate 的"自动魔法"（延迟加载、脏检查、一级缓存）在复杂业务场景下容易产生意外行为。例如，一个简单的 `article.getComments()` 可能触发 N 次额外 SQL。虽然可以通过 `@EntityGraph`、`@BatchSize` 优化，但调试成本高于直接用 SQL。

**权衡**：

- **代码量**：需要为每个实体编写 Mapper 接口和 XML（或使用 BaseMapper 继承），代码量多于 JPA 的纯注解方式。但 `MyBatis-Plus` 的 `BaseMapper<T>` 提供标准 CRUD，减少了这部分冗余。
- **数据库迁移**：MyBatis-Plus 不提供 DDL 自动生成（如 JPA 的 `ddl-auto: update`），表结构变更需要手写 SQL 脚本。项目中使用 `config/sql/init.sql` 管理 DDL。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **JPA/Hibernate** | 注解驱动，DDL 自动，JPQL | SQL 不可控，复杂查询性能调优困难 | 阅读/文章列表等查询需要精细的 SQL 控制 |
| **JOOQ** | 类型安全 SQL DSL，编译期检查 | 需要数据库元数据生成代码，学习曲线陡 | 对于 CRUD 为主的项目过于重型 |
| **JDBC Template** | 极轻量 | 缺少分页/逻辑删除/自动填充等便利功能 | 需要手写太多基础代码 |

**什么时候重新评估**：

- 如果未来引入 DDD 聚合根模式、事件溯源，评估 JPA 或 JOOQ
- 如果引入 ClickHouse/StarRocks 做分析，使用原生 JDBC 或专用 ORM

### 4.2 数据库：MySQL 8.0

| 项目 | 内容 |
|------|------|
| **选择** | MySQL 8.0 (主库 + 从库) |
| **连接池** | Druid 1.2.23 |

**为什么选择**：

1. **团队熟悉度**：MySQL 是国内团队的事实标准，运维经验丰富，备份恢复流程成熟。
2. **读写分离**：通过 binlog 主从复制实现读写分离，主库负责写入（用户注册、文章发布），从库负责查询（文章列表、评论列表），有效分散负载。`ReadWriteRoutingDataSource` 基于 Spring 事务的 `readOnly` 属性自动路由：

   ```java
   // 文件: common/src/main/java/com/kb/common/config/ReadWriteRoutingDataSource.java
   // @Transactional(readOnly = true) → SLAVE
   // @Transactional → MASTER
   ```

3. **Druid 连接池**：选择 Druid 而非更流行的 HikariCP，关键原因是 Druid 的内置 **SQL 防火墙（Wall Filter）**和全面的监控能力。在开发/测试阶段，Druid 的 SQL 监控页面能快速定位慢查询和泄露连接。

**权衡**：

- **PostgreSQL 的功能差异**：PostgreSQL 在 JSON 查询（`jsonb`）、全文搜索（`tsvector`）、窗口函数方面优于 MySQL。但项目的复杂查询主要在热搜排序（走 Redis ZSET）和文本搜索（未来可能引入 Elasticsearch），MySQL 的当前能力已足够。
- **主从延迟**：异步 binlog 复制有约 50-500ms 延迟，可能导致"刚发布的文章在列表中看不到"的体验问题。对于用户自己的操作，前端直接从 API 响应中获取新文章 ID 并导航；对于其他用户的浏览，500ms 延迟可接受。
- **Druid vs HikariCP**：HikariCP 在基准测试中比 Druid 快约 5-10%（连接获取时间）。但 Druid 的 Wall Filter 在 SQL 注入防护方面提供了额外价值，且 Druid 的 Spring Boot Starter 集成更完善。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **PostgreSQL** | 功能更丰富，ANSI SQL 更标准 | 运维经验少，工具链不熟悉 | 团队不熟悉 |
| **HikariCP** | 极致性能，Spring Boot 默认 | 无 SQL 防火墙，无内置监控页面 | 运维可观测性不足 |

**什么时候重新评估**：

- QPS > 5K 时，评估 MySQL 读写分离是否饱和，考虑增加从库数量或引入缓存层
- 需要全文搜索时，引入 Elasticsearch 而非依赖 MySQL FULLTEXT 索引
- 引入多活/异地多活时，评估 TiDB 或 CockroachDB 的分布式能力

## 5. 缓存

### 5.1 多级缓存：Redis 7 + Caffeine + Nginx

| 层级 | 技术 | TTL | 命中率 | 作用 |
|------|------|-----|--------|------|
| L0 | Nginx proxy_cache | 1s | ~85% | 边缘缓存，拦截重复查询 |
| L1 | Caffeine 本地缓存 | 1s | ~99% | JVM 内，纳秒级访问 |
| L2 | Redis ZSET | 持久 | 100% (真实数据) | 排行榜存储 + 排序 |

**为什么选择多级缓存**：

1. **热路径优化**：`GET /api/trending` 是系统最高频的接口（首页热搜榜），通过 L0/L1/L2 三级缓存，数据库完全被保护，零数据库查询。

2. **Caffeine 性能**：Caffeine 是 Java 领域性能最优的本地缓存（基于 W-TinyLFU 算法），纳秒级访问延迟。`maximumSize(1)` 设计意味着只缓存唯一的 `top10` key，内存占用极小。

3. **Redis ZSET 排行榜**：Redis Sorted Set 是唯一能高效支撑"实时排行榜"的数据结构（`ZINCRBY` O(log N)、`ZREVRANGE` O(log N + M)）。没有其他主流缓存方案能同时满足"实时更新 score"和"按 score 排序 Top K"两个需求。

**权衡**：

- **Nginx 缓存的脏读窗口**：Nginx 缓存 TTL 为 1 秒，这意味着在 1 秒内可能返回略微过期的数据。对于排行榜场景，1 秒的延迟完全可接受。
- **三级缓存一致性**：通过 Redis Pub/Sub 广播失效 Caffeine，Nginx 缓存通过 TTL 自然过期。一致性保障链为：Pub/Sub → Caffeine invalidate（即时） → Nginx TTL 过期（<= 1s） → Caffeine TTL 兜底（<= 1s）。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **仅 Redis (无本地缓存)** | 简单，一致性强 | 每次查询 ~1ms 网络延迟 | 1000 QPS 下 1ms 累加可观 |
| **Memcached** | 简单，多线程 | 无 ZSET，无法做排行榜 | ZSET 是核心需求 |
| **Hazelcast** | 分布式缓存，无额外部署 | 比 Redis+Caffeine 重，启动慢 | Docker Compose 场景下过度复杂 |

**什么时候重新评估**：

- QPS > 10K 时，评估 Redis Cluster 分片（按 articleId hash 分片，解决单 Redis 内存/带宽瓶颈）
- 热搜榜单 > Top 100 时，评估专用时序数据库或 OLAP 引擎

## 6. 消息队列

### 6.1 RabbitMQ 3

| 项目 | 内容 |
|------|------|
| **选择** | RabbitMQ 3 (management-alpine 镜像) |
| **协议** | AMQP 0-9-1 |
| **用途** | 文章发布后异步 Tag 提取 + 合规检测，点赞异步持久化 |

**为什么选择**：

1. **可靠性 + 管理 UI**：RabbitMQ 提供完善的消息确认（ACK）、持久化、死信队列机制。管理 UI（端口 15672）在开发阶段极大降低了消息流调试的难度——可以直接看到 exchange、queue、binding、消费者状态。

2. **Topic Exchange**：`article.publish` routing key 通过 Topic Exchange 灵活路由到 `tag.queue` 和 `compliance.queue`，不需要为每种消费场景创建独立的 Exchange。

3. **当前规模匹配**：日均文章发布量预估在数百至数千级别，消息量远未达到需要 Kafka 的水平。

**权衡**：

- **AMQP 协议开销**：相比 Kafka 的二进制协议，AMQP 的单条消息吞吐量低约 3-5 倍。但当前消息量（< 100 msg/s）下不构成瓶颈。
- **镜像队列（HA）**：当前单节点部署，RabbitMQ 宕机则消息积压。未来可通过镜像队列实现高可用，但镜像队列在高吞吐下性能下降明显（所有消息写入所有节点）。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Kafka** | 超高吞吐（百万 msg/s），消息持久化 + 回溯 | 运维复杂（ZK/KRaft、分区管理），部署资源需求大 | 当前规模杀鸡用牛刀 |
| **RocketMQ** | 阿里系，事务消息，延迟消息 | 运维更复杂，与 Nacos/Sentinel 版本兼容性需验证 | RabbitMQ 的管理 UI 对开发阶段更友好 |
| **Redis Streams** | 已有 Redis，零额外部署 | 消费者组管理不如 RabbitMQ 成熟，ACK 机制弱 | 点赞异步持久化用 Redis Stream（`like:events`），但 Tag/Compliance 用 RabbitMQ |

**什么时候重新评估**：

- 消息量 > 10K msg/s 时，迁移至 Kafka
- 事件溯源需求（需要回溯历史消息）时，迁移至 Kafka
- 事务消息需求时，评估 RocketMQ

## 7. 安全

### 7.1 JWT：jjwt 0.12.6

| 项目 | 内容 |
|------|------|
| **选择** | jjwt (io.jsonwebtoken) 0.12.6 |
| **算法** | HMAC-SHA256 |
| **结构** | jjwt-api + jjwt-impl + jjwt-jackson |

**为什么选择**：

1. **轻量无依赖**：jjwt 是一个纯粹 JWT 规范的实现，不绑定 Spring Security，不引入额外的安全框架依赖。项目使用自建的 `JwtAuthFilter`（Gateway Filter），无需引入整个 Spring Security 体系。

2. **API 简洁**：Token 生成和解析只需几行代码：

   ```java
   // 文件: common/src/main/java/com/kb/common/utils/JwtUtils.java
   Jwts.builder().subject(userId).signWith(key).expiration(date).compact();
   Jwts.parser().verifyWith(key).build().parseSignedClaims(token);
   ```

3. **Jackson 集成**：`jjwt-jackson` 使用 Jackson 进行 JSON 序列化（与项目已有 Jackson 依赖一致），避免了引入额外的 JSON 库。

**权衡**：

- **无内置刷新 Token 机制**：需要自建 Token 刷新逻辑。当前项目使用无状态 JWT，过期后需重新登录。适合当前场景（单页面应用，用户活跃时 Token 自动有效）。
- **无 OAuth2/OIDC**：不支持标准化的第三方登录（微信、GitHub）。如果未来需要，需要切换到 Spring Security OAuth2。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Nimbus JOSE + JWT** | 最完整的 JWT/JOSE 实现，OAuth2 推荐 | 依赖重，API 更复杂 | 功能超出当前需求 |
| **auth0-java** | Auth0 官方 | API 灵活性差，与 Spring 集成需额外配置 | 不如 jjwt 灵活 |

**什么时候重新评估**：

- 需要第三方登录时，引入 Spring Security OAuth2 Client + Nimbus
- 需要 Token 刷新机制时，扩展 JwtUtils 或评估 OAuth2

## 8. AI 集成

### 8.1 流式调用：Spring WebFlux + SSE

| 项目 | 内容 |
|------|------|
| **选择** | Spring WebFlux (Reactor Netty) + Server-Sent Events |
| **熔断** | Resilience4j ReactiveCircuitBreaker |
| **连接池** | Reactor Netty ConnectionProvider |

**为什么选择**：

1. **非阻塞 I/O**：调用 LLM API 是一个典型的长等待 I/O 场景（TTFT 500ms-2s，完整生成 5-30s）。WebFlux 的 Reactor Netty 使用事件循环（event loop）而非线程池，少量线程即可管理数百个并发的 LLM 连接。如果使用 Servlet 同步模型，每个连接占用一个线程，100 个并发就需要 100 个线程（栈内存约 100MB）。

2. **Reactive Streaming 原生支持**：`Flux<ServerSentEvent<String>>` 是 Spring WebFlux 的一等公民。从 OpenAI API 的 `stream=true` 响应（逐行 SSE）到浏览器 EventSource，整个链路归一到 Reactive Streams 规范，无需手动管理背压和线程切换。

3. **Resilience4j 集成**：`ReactiveCircuitBreaker` 原生理解 Reactor 的信号（`onError`、`onComplete`），熔断状态切换与响应式流无缝衔接。

**权衡**：

- **调试复杂度**：响应式堆栈追踪（`reactor.core.publisher.FluxOnAssembly`）比同步调用难以阅读。需要配置 `Hooks.onOperatorDebug()` 或在 IDE 中使用 Reactor 调试插件。
- **Demo 模式必要性**：依赖外部 LLM API 意味着开发和测试环境必须配置 API Key 或有替代方案。Demo 模式设计的额外投入是必要的。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Servlet Async + DeferredResult** | Servlet 生态，易调试 | 线程模型仍是"每请求一线程"，背压处理复杂 | 并发能力弱于 WebFlux |
| **WebSocket** | 双向通信 | AI 续写场景不需要客户端→服务器推送，EventSource 不支持自定义头（JWT 需用 URL 参数） | 方向不匹配 |

**什么时候重新评估**：

- 引入 SSE 双向通信需求（如实时评论通知）时，评估 WebSocket（SockJS + STOMP）
- 接入多个 AI 提供商（OpenAI + Claude + 本地模型）时，评估引入 AI Gateway 层（如 LiteLLM）

## 9. 前端

### 9.1 React 19 + Vite 8 + TypeScript 6

| 项目 | 内容 |
|------|------|
| **选择** | React 19 + Vite 8 + TypeScript 6 |
| **状态管理** | Zustand 5 |
| **路由** | React Router 7 |
| **HTTP** | Axios 1.18 |
| **SSE** | @microsoft/fetch-event-source |

**为什么选择**：

1. **Vite 速度快**：Vite 使用原生 ESM 开发服务器，冷启动时间 < 1 秒（CRA/Webpack 需要 10-30 秒）。HMR（热模块替换）几乎即时生效，极大提升前端开发体验。

2. **React 生态**：React 19 引入 Server Components 和 Actions，虽然当前仅使用客户端渲染（CSR），但为未来 SSR 演进预留了空间。Zustand 作为轻量级状态管理（< 1KB），比 Redux 更简洁。

3. **TypeScript 6**：类型系统减少运行时错误，与后端 Java 强类型风格一致。`@microsoft/fetch-event-source` 提供比浏览器原生 EventSource 更好的 SSE 客户端（支持 POST 请求、自定义请求头）。

**权衡**：

- **CSR 的 SEO**：React SPA 默认客户端渲染，搜索引擎可能无法抓取内容。知识平台是登录后可用的创作工具，SEO 需求不强。
- **包体积**：React + ReactDOM 约 130KB gzipped，加上路由、状态管理等约 200KB。首次加载约 1-2 秒（Nginx gzip 压缩），可接受。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Vue 3 + Vite** | 学习曲线平缓，中文文档好 | 同等优质，选择 React 纯因团队偏好 | 团队更熟悉 React |
| **Angular** | 企业级全家桶 | 学习曲线陡，包体积大，对小项目过重 | 过度设计 |
| **Svelte** | 编译时框架，极小程序体积 | 生态较小，成熟度不及 React | 生态风险 |

## 10. 容器化与部署

### 10.1 Docker Compose

| 项目 | 内容 |
|------|------|
| **选择** | Docker Compose (compose.yml, 12 containers) |
| **网络** | bridge: kb-network |
| **存储** | named volumes (mysql_data, redis_data 等) |

**为什么选择**：

1. **简单编排**：`docker-compose up -d` 一条命令启动整套环境，开发/测试/演示环境配置完全一致。无需学习 K8s 的 Pod/Service/Deployment/Ingress/ConfigMap 等概念。

2. **单机够用**：当前所有服务（8 个 Java + MySQL + Redis + RabbitMQ + Nacos + Nginx）约 12 个容器，单台开发服务器（16GB RAM）完全承载。

3. **开发-生产一致性**：使用相同的 Dockerfile 和 compose 文件（通过 profile 控制），避免了"开发环境正常，生产环境异常"的问题。

**权衡**：

- **无自愈能力**：容器崩溃后 Docker Compose 默认不自动重启（需配置 `restart: always`），不能像 K8s 那样自动调度到健康节点。
- **无滚动更新**：更新服务需要 `docker-compose up -d --build`，有短暂停机。当前开发/演示阶段可接受。

**替代方案对比**：

| 方案 | 优点 | 缺点 | 为什么不选 |
|------|------|------|-----------|
| **Kubernetes** | 自愈、滚动更新、自动扩缩容 | 学习曲线陡，本地开发需要 minikube/kind，运维成本高 | 当前规模过度 |
| **Docker Swarm** | 比 K8s 简单，Docker 原生 | 社区萎缩，生态不如 K8s | 不如直接学习 K8s 为未来做准备 |

**什么时候重新评估**：

- 服务实例数 > 3/服务 时，迁移至 K8s（详见《08-生产环境演进方案》）
- 需要多机部署、自愈、自动扩缩容时，迁移至 K8s

4. **可观测**：Druid 监控、RabbitMQ 管理 UI、Nacos 控制台——开发阶段可观测性优先于极致性能
