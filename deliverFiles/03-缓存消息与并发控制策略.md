## 1. 概述

知识平台在核心业务路径上引入了三层加速与解耦机制：**Redis 作为共享状态存储与消息通道**、**Caffeine 作为本地热点缓存**、**RabbitMQ 作为异步任务总线**。三者协同形成了从用户请求到最终持久化的完整数据链路。

本文档覆盖以下七个主题领域：

| 领域 | 核心机制 | 涉及组件 |
|------|---------|---------|
| 热度榜单 | Redis ZSET + Pub/Sub + Caffeine 三级缓存 | article-service |
| 读写去重 | Redis String SETNX 幂等控制 | article-service, interact-service |
| 点赞持久化 | Redis Stream 消费组 + JDBC 批量写入 | interact-service |
| 评论持久化 | Redis List + @Scheduled 定时批量入库 | interact-service |
| 异步任务分发 | RabbitMQ TopicExchange 扇出 | article-service, tag-extract-service, compliance-service |
| 内容审核 | 消息消费 + 规则引擎模拟 AI | compliance-service |
| 标签提取 | 消息消费 + 分词匹配模拟 AI | tag-extract-service |

## 3. Redis 数据结构设计

系统使用 7 种 Redis 数据结构，每种承担不同的业务职责：

### 3.1 数据结构总览表

| # | 类型 | Key 模式 | 生命周期 | 核心操作 | 业务目的 |
|---|------|---------|---------|---------|---------|
| 1 | ZSET | `hot_articles` | 永久（衰减清理） | ZINCRBY, ZREVRANGE WITHSCORES | 热度排行榜 |
| 2 | String | `read:article:{id}:user:{uid}` | TTL 300s | SETNX, GET | 阅读去重 |
| 3 | String | `like:article:{id}:user:{uid}` | 永久 | SETNX | 点赞去重 |
| 4 | String | `token:blacklist:{token}` | TTL=JWT 剩余有效期 | SET (logout), hasKey (gateway) | Token 黑名单 |
| 5 | Stream | `like:events` | 持久（消费后不删） | XADD, XREADGROUP, XACK | 点赞异步持久化 |
| 6 | List | `comment:events` | 瞬时（消费即出队） | LPUSH, RPOP | 评论异步持久化 |
| 7 | Pub/Sub | `hot_articles:refresh` | 瞬时（无持久化） | PUBLISH, SUBSCRIBE | 跨实例缓存失效通知 |

### 3.2 热度 ZSET (`hot_articles`)

**数据结构**：Sorted Set，member 为 `articleId`（字符串），score 为热度分（double）。

**分数规则**：

| 操作 | 增量 | 说明 |
|------|------|------|
| 阅读 | +1 | 同一用户 300s 内重复阅读不计入（通过 `read:article:{id}:user:{uid}` 去重） |
| 点赞 | +3 | 点赞权重高于阅读，体现强信号 |
| 评论 | +1 | 评论权重与阅读持平 |
| 每日衰减 | x 0.9 | 每日 3:00 AM 执行，X 低于 1.0 自动移除 |

**查询方式**：

```
ZREVRANGE hot_articles 0 9 WITHSCORES   # 取 Top 10 及分数
```

**实现参考**：
- 文件：`article-service/.../service/impl/HotArticleServiceImpl.java` —— recordRead() / recordLike() / recordComment() 分别执行 ZINCRBY
- 文件：`interact-service/.../service/impl/LikeServiceImpl.java` —— likeArticle() 中调用 ZINCRBY +3（步骤 3）

### 3.3 去重 Key（String 类型）

#### 阅读去重

```
Key:    read:article:{articleId}:user:{userId}
Value:  "1"
TTL:    300 秒
操作:   SETNX（原子判断-设置）
```

逻辑：`HotArticleServiceImpl.recordRead()` 调用 `setIfAbsent(dedupKey, "1", Duration.ofSeconds(300))`，仅当 Key 不存在时返回 true，此时才执行 `ZINCRBY +1`。

#### 点赞去重

```
Key:    like:article:{articleId}:user:{userId}
Value:  "1"
TTL:    无（永久）
操作:   SETNX
```

逻辑：`LikeServiceImpl.likeArticle()` 中步骤 1 执行 `setIfAbsent(dedupKey, "1")`，若返回 false 则抛出 `BusinessException("您已点赞过该文章")`。

**事务回滚**：如果后续 Stream 写入失败（步骤 2 异常），会执行 `redisTemplate.delete(dedupKey)` 回滚去重 Key，保证用户可重试。

#### Token 黑名单

```
Key:    token:blacklist:{token字符串}
Value:  任意非空
TTL:    JWT 剩余有效期（exp - now）
操作:   SET (用户登出时), EXISTS/hasKey (网关每次请求检查)
```

网关 `JwtAuthFilter` 使用 `ReactiveRedisTemplate.hasKey()` 检查黑名单。登出接口设置 key 时同步计算 TTL，保证黑名单条目不会永久占用内存。

### 3.4 点赞事件流（Redis Stream）

**Key**：`like:events`
**消费组**：`like-persist-group`
**消费者**：`like-persist-consumer`

**消息格式**（Map）：

```
{
  "articleId": "10001",
  "userId":    "42",
  "timestamp": "1751731200000"
}
```

**生产端**（`LikeServiceImpl.likeArticle()` 步骤 2）：

```java
Map<String, String> event = new HashMap<>();
event.put("articleId", String.valueOf(articleId));
event.put("userId", String.valueOf(userId));
event.put("timestamp", String.valueOf(System.currentTimeMillis()));
redisTemplate.opsForStream().add(RedisKeyConstants.LIKE_EVENTS_STREAM, event);
```

**消费端**（`LikePersistConsumer`）：

- 实现 `InitializingBean`，在 `afterPropertiesSet()` 中创建消费组（若不存在）并启动 daemon 线程
- 每次轮询最多拉取 50 条消息（`StreamReadOptions.count(50)`）
- 逐条执行 SQL：`UPDATE articles SET like_count = like_count + 1 WHERE id = ?`
- 成功处理的消息调用 `XACK` 确认
- 处理失败的消息不确认，自然进入重试（at-least-once 语义）

**文件**：`interact-service/.../consumer/LikePersistConsumer.java`

### 3.5 评论事件列表（Redis List）

**Key**：`comment:events`
**操作**：`LPUSH` 写入（生产端）、`RPOP` 批量取出（消费端）

**消息格式**（JSON 字符串）：

```json
{
  "articleId": "10001",
  "userId": "42",
  "content": "写得很好，受益匪浅！",
  "createdAt": "2026-07-06T10:30:00"
}
```

**消费端**（`CommentPersistConsumer`）：

- `@Scheduled(fixedDelay = 5000)` 每 5 秒执行一次
- 每次批量 `RPOP` 最多 100 条
- 通过 MyBatis-Plus `commentMapper.insert(comments)` 批量写入
- 5 秒的持久化间隙意味着：评论写入 Redis List 后最迟 5 秒才能在其他实例查询到

**文件**：`interact-service/.../consumer/CommentPersistConsumer.java`

### 3.6 缓存刷新频道（Redis Pub/Sub）

**Channel**：`hot_articles:refresh`

**发布时机**（所有修改 `hot_articles` ZSET 的操作完成后均发布）：

| 触发场景 | 发布消息 | 调用位置 |
|---------|---------|---------|
| 用户阅读文章 | `read:{articleId}` | `HotArticleServiceImpl.recordRead()` |
| 用户点赞 | `like:{articleId}` | `LikeServiceImpl.publishRefresh()` |
| 用户评论 | `comment:{articleId}` | 评论 Service |
| 每日衰减 | `decay` | `HeatDecayScheduler.decayScores()` |

**订阅端**（`HotArticlesRefreshListener`）：

- 实现 `MessageListener.onMessage()`
- 收到任意消息后调用 `cacheManager.invalidate()` 失效本地 Caffeine 缓存
- 订阅通过 `RedisMessageListenerContainer` 在独立线程中运行

**关键文件**：
- 配置：`article-service/.../config/RedisPubSubConfig.java`
- 监听器：`article-service/.../listener/HotArticlesRefreshListener.java`

## 4. Caffeine 本地缓存层

### 4.1 缓存配置

**文件**：`article-service/.../cache/HotArticlesCacheManager.java`

```java
Caffeine.newBuilder()
    .expireAfterWrite(Duration.ofSeconds(cacheTtlSeconds))  // 默认 1s
    .maximumSize(1)                                          // 只缓存 Top 10 列表
    .recordStats()                                           // 开启统计
    .build();
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| expireAfterWrite | 1 秒 | 写后过期时间，可通过 `hot-articles.cache-ttl-seconds` 配置 |
| maximumSize | 1 | 仅缓存一个 key（"top10"），内存极低 |
| recordStats | true | 记录 hit/miss 统计，可通过 Actuator 暴露 |

### 4.2 缓存操作流程

```
getTop10()
  │
  ▼
cacheManager.getOrCompute(supplier)
  │
  ├── 命中 ──▶ 直接返回本地缓存 (L1 hit, 0ms)
  │
  └── 未命中 ──▶ supplier.get() 执行
                   │
                   ├── ZREVRANGE hot_articles 0 9 WITHSCORES   (L2: Redis ZSET)
                   │
                   └── 批量 SELECT FROM articles WHERE id IN (...)  (L3: MySQL)
                       │
                       └── 组装 TopArticleVO 列表 → 写入 Caffeine → 返回
```

### 4.3 缓存失效机制

| 方式 | 触发条件 | 延迟 |
|------|---------|------|
| Pub/Sub 主动失效 | Redis 发布 `hot_articles:refresh` 消息 | 毫秒级（网络 RTT） |
| 定时衰减被动失效 | `HeatDecayScheduler.decayScores()` 执行后 | 固定 3:00 AM |
| TTL 自动过期 | 写后 1 秒自然过期 | <= 1 秒 |

三级保障确保：即使 Pub/Sub 消息丢失（网络分区等极端情况），缓存也会在 1 秒后自动刷新，不会出现长期脏数据。

### 4.4 缓存命中率估算

- **TTL 1 秒**：在 1 秒内所有并发请求共享同一份缓存
- **QPS 1000 场景**：1 秒内 1000 个请求，仅 1 次穿透到 Redis+MySQL，命中率 99.9%
- **maxSize=1**：内存占用约 2-5 KB（10 条 TopArticleVO），可忽略

## 5. RabbitMQ 消息队列

### 5.1 拓扑结构

```
                          article.topic.exchange
                          (TopicExchange, durable)
                                   │
            ┌──────────────────────┼──────────────────────┐
            │ article.tag         │ article.publish       │ article.compliance
            ▼                      ▼                       ▼
   ┌─────────────────┐    ┌─────────────────┐    ┌─────────────────────┐
   │ article.tag.queue│    │ (routed to      │    │ article.compliance. │
   │ (durable)       │    │  both queues)   │    │ queue (durable)     │
   │ Consumer:       │    │                 │    │ Consumer:           │
   │ TagExtract      │    │                 │    │ ComplianceCheck     │
   │ (5 threads)     │    │                 │    │ (5-10 threads)      │
   └─────────────────┘    └─────────────────┘    └─────────────────────┘
```

**设计要点**：

- `article.publish` 路由键同时绑定到两个队列，实现**类扇出**（fan-out）效果：发布一篇文章同时触发标签提取和合规检测
- 两个消费者各自独立消费，互不阻塞
- 标签提取 `article.tag` 路由键作为独立绑定，支持未来单独发送标签重提取消息

### 5.2 交换机与队列声明

**文件**：`article-service/.../config/RabbitMQConfig.java`

```java
public static final String TOPIC_EXCHANGE       = "article.topic.exchange";
public static final String TAG_QUEUE            = "article.tag.queue";
public static final String COMPLIANCE_QUEUE     = "article.compliance.queue";
public static final String TAG_ROUTING_KEY      = "article.tag";
public static final String COMPLIANCE_ROUTING_KEY = "article.compliance";
public static final String PUBLISH_ROUTING_KEY  = "article.publish";
```

**绑定关系**：

| 队列 | 绑定 Routing Key | 说明 |
|------|-----------------|------|
| `article.tag.queue` | `article.tag`, `article.publish` | 标签提取 |
| `article.compliance.queue` | `article.compliance`, `article.publish` | 合规检测 |

### 5.3 消息格式

**发送端**（`ArticleServiceImpl.publish()`）：

```java
Map<String, Object> message = new HashMap<>();
message.put("articleId", article.getId());
message.put("title", article.getTitle());
message.put("content", article.getContent());
message.put("authorId", article.getAuthorId());
message.put("publishedAt", LocalDateTime.now().toString());

rabbitTemplate.convertAndSend(
    RabbitMQConfig.TOPIC_EXCHANGE,
    "article.publish",
    message);
```

**序列化**：`Jackson2JsonMessageConverter`，消息体为 JSON。

**消息字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| articleId | Long/Number | 文章 ID |
| title | String | 文章标题 |
| content | String | 文章正文 |
| authorId | Long/Number | 作者 ID |
| publishedAt | String | 发布时间（ISO_LOCAL_DATE_TIME） |

### 5.4 可靠性配置

| 配置项 | 值 | 说明 |
|-------|-----|------|
| 交换机持久化 | durable=true | 服务重启不丢失交换机定义 |
| 队列持久化 | durable=true | 服务重启不丢失队列及消息 |
| 发布确认 | publisher-confirm-type: correlated | 异步确认每条消息是否到达 Broker |
| 发布退回 | publisher-returns: true | 无法路由的消息回调通知 |
| 消费确认 | MANUAL | 消费者手动 ACK/NACK |
| 失败处理 | basicNack(deliveryTag, false, false) | 不重新入队，避免毒消息循环 |

## 7. 合规检测消费逻辑

**文件**：`compliance-service/.../consumer/ComplianceCheckConsumer.java`

### 7.1 三态审核模型

```
                   ┌─────────────────────┐
                   │  文章内容 (title +   │
                   │  content.toLowerCase)│
                   └──────────┬──────────┘
                              │
                   ┌──────────▼──────────┐
                   │  content.length <    │
                   │  10 ?                │
                   └──────┬──────┬───────┘
                     YES  │      │  NO
                          ▼      │
                    ┌──────────┐ │
                    │  REVIEW  │ │
                    │ (内容过短)│ │
                    └──────────┘ │
                                 ▼
                   ┌─────────────────────┐
                   │  是否包含敏感词？    │
                   │  违规/违法/赌博/色情 │
                   │  暴力/诈骗/传销/毒品 │
                   │  枪支/走私           │
                   └──────┬──────┬───────┘
                     YES  │      │  NO
                          ▼      │
                    ┌──────────┐ │
                    │  BLOCK   │ │
                    │ set deleted=1│
                    │ audit_result=│
                    │ "BLOCK"  │ │
                    └──────────┘ │
                                 ▼
                   ┌─────────────────────┐
                   │  是否包含可疑词？    │
                   │  广告/推广/联系方式  │
                   │  加微信/QQ群/免费领取│
                   │  点击链接/代理/返利  │
                   └──────┬──────┬───────┘
                     YES  │      │  NO
                          ▼      ▼
                    ┌──────┐  ┌──────┐
                    │REVIEW│  │ PASS │
                    └──────┘  └──────┘
```

### 7.2 审核结果处理

| 结果 | 操作 | 状态说明 |
|------|------|---------|
| PASS | 仅设置 `audit_result = "PASS"` | 文章正常展示 |
| REVIEW | 设置 `audit_result = "REVIEW"` | 文章正常展示但标记需人工复核 |
| BLOCK | 设置 `audit_result = "BLOCK"` + `deleted = 1` | 逻辑删除，前端不可见 |

### 7.3 并发配置

- `concurrency = "5-10"`：最小 5、最大 10 个并发消费者
- 合规检测涉及外部 API 调用（生产环境），IO 密集型
- 当前为模拟实现（关键词匹配），实际部署时应替换为专业审核服务

### 7.4 安全性说明

当前敏感词库为演示用简化版本（10 个敏感词 + 9 个可疑词）。生产环境应：

1. 接入专业内容审核服务（阿里云内容安全、网易易盾等）
2. 支持图片、视频等多模态检测
3. 敏感词配置外部化（Nacos Config / Apollo）
4. 审核日志持久化至独立表，便于审计追溯

## 8. 热度衰减调度

**文件**：`article-service/.../scheduler/HeatDecayScheduler.java`

### 8.1 设计目的

纯热度累积会导致旧文章长期占据榜单（"马太效应"），新文章无法上榜。每日衰减模拟了"时间衰减"的自然规律。

### 8.2 执行逻辑

```
每日 3:00 AM (Cron: 0 0 3 * * ?)
  │
  ▼
ZRANGE hot_articles 0 -1 WITHSCORES     # 获取全部热度条目（O(N)）
  │
  ▼
for each (member, score):
    newScore = score × 0.9              # 衰减因子 0.9
    │
    ├── newScore < 1.0 ?               # 低于最低阈值
    │   └── ZREM hot_articles member   # 从排行榜移除
    │
    └── newScore >= 1.0 ?
        └── ZADD hot_articles member newScore  # 覆盖旧分数
  │
  ▼
cacheManager.invalidate()               # 失效本地缓存
PUBLISH hot_articles:refresh "decay"    # 通知其他实例
```

### 8.3 配置参数

| 参数 | 默认值 | 配置键 | 说明 |
|------|--------|--------|------|
| 衰减因子 | 0.9 | `hot-articles.decay-factor` | 每次衰减保留 90% 热度 |
| 最低分数 | 1.0 | `hot-articles.decay-min-score` | 低于此值从排行榜移除 |
| 执行时间 | 每日 3:00 AM | `hot-articles.decay-cron` | Cron 表达式，可定制 |

### 8.4 性能估算

- 时间复杂度：O(N)，N 为 ZSET 成员数
- 正常运营下 N < 5000，执行时间 < 1 秒
- 执行在低峰期（3:00 AM），对业务无感知

## 9. 并发控制与数据一致性

### 9.1 多层级保护策略

```
┌─────────────────────────────────────────────────────────────┐
│  层级            │  机制                  │  保护目标       │
├─────────────────────────────────────────────────────────────┤
│  数据库层        │  Atomic UPDATE         │  点赞数准确累加 │
│                  │  MyBatis-Plus 逻辑删除 │  软删除一致性   │
├─────────────────────────────────────────────────────────────┤
│  Redis 层        │  SETNX 原子操作        │  点赞/阅读去重  │
│                  │  ZINCRBY 原子递增      │  热度分并发安全 │
│                  │  Stream 消费组         │  至少一次投递   │
├─────────────────────────────────────────────────────────────┤
│  应用层          │  Manual ACK            │  消息可靠消费   │
│                  │  Transaction Rollback  │  Stream 写入失败│
│                  │                        │  回滚去重 Key   │
├─────────────────────────────────────────────────────────────┤
│  缓存层          │  Pub/Sub 主动失效      │  跨实例一致性   │
│                  │  TTL 兜底              │  最终一致性     │
└─────────────────────────────────────────────────────────────┘
```

### 9.2 点赞流程的并发安全分析

点赞业务是最复杂的并发场景，涉及多个存储系统的状态变更：

```
likeArticle(articleId, userId)
  │
  ├── 1. SETNX like:article:{id}:user:{uid} "1"     # Redis 原子去重
  │   ├── 返回 false → BusinessException("已点赞")  # 幂等：同一个 userId 只允许一次
  │   └── 返回 true  → 继续
  │
  ├── 2. XADD like:events {articleId, userId, timestamp}  # Redis Stream 写入
  │   └── 异常 → DELETE dedupKey (回滚!) + throw Exception  # 保证可重试
  │
  ├── 3. ZINCRBY hot_articles {articleId} 3          # 热榜 +3（原子递增）
  │   └── 异常 → log.warn()  # 热度更新失败不影响主流程
  │
  └── 4. PUBLISH hot_articles:refresh "like:{id}"     # 广播缓存失效
```

**关键并发保证**：

1. **SETNX** 保证同一用户只能点赞一次（即使并发请求，只有第一个成功）
2. **ZINCRBY** 是 Redis 原子操作，多客户端并发调用结果正确
3. **Stream 消费组** 保证同一条消息不会被不同消费者重复处理
4. **回滚机制**：如果 Step 2 失败，立即删除 Step 1 创建的 Key，避免"去重成功但点赞未记录"的永久不一致状态

### 9.3 未使用分布式锁的原因

在本系统设计中，以下原因使分布式锁不是必需品：

| 场景 | 替代方案 | 效果 |
|------|---------|------|
| 点赞去重 | Redis SETNX | 单 Key 原子性，等价于分布式锁但更轻量 |
| 热度递增 | Redis ZINCRBY | 自带原子递增，无需额外同步 |
| 点赞数持久化 | MySQL `UPDATE ... SET like_count = like_count + 1` | 数据库原子操作 |
| 内容审核结果 | RabbitMQ 单队列单消费者 | 天然串行，或 MyBatis-Plus updateById 乐观锁 |

如果未来需要强一致性的复杂写操作（如"用户积分 + 热度更新 + 通知推送"三合一事务），可考虑引入 Redisson 分布式锁或 Redis Lua 脚本。

    publisher-confirm-type: correlated

### 10.3 Nginx 代理缓存配置

```nginx
# config/nginx/nginx.conf
proxy_cache_path /tmp/nginx_cache_trending
    levels=1:2
    keys_zone=trending_cache:10m
    max_size=50m
    inactive=5m
    use_temp_path=off;

location /api/trending {
    proxy_pass http://gateway_cluster;
    proxy_cache trending_cache;
    proxy_cache_valid 200 1s;
    proxy_cache_use_stale error timeout updating;
    proxy_cache_lock on;
    proxy_cache_lock_timeout 3s;
    add_header X-Cache-Status $upstream_cache_status;
}
```

这构成了 **L0/L1/L2 三级缓存架构**：
- **L0**：Nginx HTTP 缓存（1s TTL，合并并发请求）
- **L1**：Caffeine 本地缓存（1s TTL，进程内最快）
- **L2**：Redis ZSET + MySQL（分布式数据源）

## 11. 运维监控

### 11.1 关键指标

| 指标 | 采集方式 | 告警阈值建议 |
|------|---------|-------------|
| Caffeine 命中率 | `cache.stats().hitRate()` 通过 Actuator | < 95% |
| Redis ZSET 大小 | `ZCARD hot_articles` | > 10000 需关注 |
| Stream PEL 堆积 | `XPENDING like:events like-persist-group` | > 1000 条 |
| List 长度 | `LLEN comment:events` | > 5000 条 |
| RabbitMQ 队列深度 | Management API | > 1000 条 |
| 热度衰减耗时 | 日志中 `耗时={}ms` | > 5000ms |

### 11.2 排查指引

| 现象 | 可能原因 | 检查步骤 |
|------|---------|---------|
| 热榜不更新 | Caffeine TTL 过长 / Pub/Sub 订阅断开 | 检查 `ZCARD hot_articles`，确认 Caffeine stats hitRate |
| 点赞数不增长 | LikePersistConsumer 停止 / Stream 消息堆积 | 检查 `XPENDING like:events like-persist-group` |
| 标签不显示 | TagExtractConsumer 异常 / RabbitMQ 连接断开 | 检查 RabbitMQ 管理界面队列状态 |
| 合规检测失效 | ComplianceCheckConsumer 异常 | 检查日志 `合规检测异常` 关键词 |
| 热榜数据陈旧 | 衰减停止 / Pub/Sub 丢消息 | 确认 `HeatDecayScheduler` 调度日志，检查 TTL 兜底是否生效 |
