## 1. 概述

AI 助手服务（`ai-assistant-service`）提供文章流式续写能力。用户在前端编辑器中输入上下文内容，系统通过 **Server-Sent Events (SSE)** 协议逐字返回 AI 生成的续写文本，实现类似 ChatGPT 的打字机效果。

### 1.1 核心特性

| 特性 | 实现方式 |
|------|---------|
| 流式输出 | SSE (text/event-stream)，逐块推送 |
| 协议兼容 | OpenAI Chat Completions API 兼容（支持 DeepSeek / Moonshot / Ollama 等） |
| 熔断降级 | Resilience4j Circuit Breaker，失败率 > 50% 自动熔断 |
| 心跳保活 | 每 15 秒发送 SSE 注释 `:keepalive` |
| Demo 模式 | 无 API Key 时使用本地模拟输出，方便开发与演示 |
| 反应式栈 | Spring WebFlux + Reactor Netty，非阻塞全链路 |

### 1.2 适用场景

- 文章编辑器中，用户写了开头，AI 续写后续段落
- 内容创作辅助，如博客、技术文档的长文生成
- 可作为通用 AI 文本生成网关的基础，扩展支持摘要、翻译等任务

## 4. SSE 协议与接口设计

### 4.1 Server-Sent Events 简介

SSE 是一种基于 HTTP 的单向流式通信协议。与 WebSocket 不同，SSE 是纯 HTTP，天然兼容所有 HTTP 代理和负载均衡器。对于 AI 文本生成这类"服务端推送、客户端只读"的场景，SSE 比 WebSocket 更合适。

### 4.2 接口定义

**文件**：`ai-assistant-service/src/main/java/com/kb/ai/controller/AiContinueController.java`

```
POST /api/ai/continue
Content-Type: application/json
X-User-Id: {userId}  (可选，默认 0)

请求体:
{
  "context": "人工智能技术近年来发展迅速，特别是大语言模型的出现..."  // 1-4000 字，必填
}

响应:
Content-Type: text/event-stream

data: 让人

data: 工智能

data: 在多个

...
data: [DONE]
```

### 4.3 请求体 DTO

**文件**：`ai-assistant-service/src/main/java/com/kb/ai/dto/ContinueRequest.java`

```java
@Data
public class ContinueRequest {
    @NotBlank(message = "上下文内容不能为空")
    @Size(max = 4000, message = "上下文内容不能超过 4000 字")
    private String context;
}
```

### 4.4 SSE 数据格式

每个 `data:` 行包含 AI 生成的一个文本片段。片段粒度由 LLM 流式响应决定（通常 1-4 个字符）。客户端 `EventSource` 或自定义 `fetch` 可逐行解析。

**完整示例**：

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

data: 让我来为

data: 你展开

data: 分析这个

data: 问题。

:keepalive

data: 从多个

data: 角度来看

...

data: [DONE]
```

**`[DONE]` 标记**：模仿 OpenAI 流式协议，表示生成结束。客户端收到此标记后应关闭连接。

**`:keepalive` 行**：SSE 注释（以冒号开头），浏览器 EventSource 自动忽略，但保持 TCP 连接活跃，防止代理或网关因超时断开连接。

### 4.5 控制器生命周期日志

```java
return aiStreamService.continueStream(request.getContext(), userId)
    .doOnComplete(() -> log.info("AI 续写完成: userId={}", userId))
    .doOnCancel(()   -> log.info("客户端断开连接: userId={}", userId))
    .doOnError(e     -> log.error("AI 续写异常: userId={}, error={}", userId, e.getMessage()));
```

三种终止方式均有日志记录，便于监控和排查。

## 5. 核心实现：OpenAiStreamServiceImpl

**文件**：`ai-assistant-service/src/main/java/com/kb/ai/service/impl/OpenAiStreamServiceImpl.java`

### 5.1 初始化流程

```java
@PostConstruct
public void init() {
    this.webClient = WebClient.builder()
        .baseUrl(aiConfig.getBaseUrl())
        .defaultHeader("Authorization", "Bearer " + aiConfig.getApiKey())
        .defaultHeader("Content-Type", "application/json")
        .codecs(config -> config.defaultCodecs().maxInMemorySize(512 * 1024))
        .build();

    this.circuitBreaker = circuitBreakerFactory.create("openai-api");
}
```

**WebClient 连接池**（Reactor Netty 默认配置）：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| maxConnections | 100 | 最大连接数 |
| pendingAcquireTimeout | 10s | 等待获取连接的超时 |
| maxIdleTime | 60s | 最大空闲时间 |

### 5.2 模式选择

```java
public Flux<ServerSentEvent<String>> continueStream(String context, Long userId) {
    if (aiConfig.isDemoMode() || aiConfig.getApiKey() == null || aiConfig.getApiKey().isBlank()) {
        log.info("使用 Demo 模式: userId={}", userId);
        return demoStream();
    }

    return circuitBreaker.run(
        callOpenAiStream(context),
        throwable -> fallbackStream(throwable)
    );
}
```

自动降级逻辑：无 API Key 或无 Demo Mode 配置时，自动使用模拟输出，无需手动切换。

### 5.3 OpenAI API 调用

**System Prompt**（内置中文写作助手指令）：

```
你是一个专业的内容创作助手。请根据用户提供的上文内容，
自然地续写后续段落。续写内容应：
1. 保持与上文一致的语气和风格
2. 逻辑连贯，内容有深度
3. 使用中文书写（除非上文是其他语言）
```

**请求体构造**：

```java
Map.of(
    "model", aiConfig.getModel(),
    "messages", List.of(
        Map.of("role", "system", "content", SYSTEM_PROMPT),
        Map.of("role", "user", "content", context)
    ),
    "stream", true,
    "max_tokens", aiConfig.getMaxTokens(),
    "temperature", aiConfig.getTemperature()
);
```

**SSE 解析管道**（Reactor Flux 操作链）：

```
webClient.post()
    .uri("/v1/chat/completions")
    .bodyValue(requestBody)
    .accept(TEXT_EVENT_STREAM)
    .retrieve()
    .bodyToFlux(String.class)                    // Step 1: 逐行读取响应体
    .filter(line -> line.startsWith("data:"))    // Step 2: 过滤非 data 行
    .map(line -> line.substring(5).trim())       // Step 3: 去掉 "data:" 前缀
    .filter(data -> !data.isBlank()              // Step 4: 过滤空行和结束标记
                 && !"[DONE]".equals(data))
    .map(this::extractContent)                   // Step 5: JSON → 文本
    .filter(token -> !token.isEmpty())           // Step 6: 过滤空 token
    .map(token -> ServerSentEvent.<String>       // Step 7: 包装为 SSE
         builder().data(token).build())
    .mergeWith(heartbeat())                       // Step 8: 合并心跳流
    .timeout(Duration.ofSeconds(120))             // Step 9: 超时保护
    .doOnCancel(() -> log.debug("上游 LLM 请求取消")) // Step 10: 取消回调
```

### 5.4 Delta Content 提取

OpenAI 流式响应的每个 `data:` 行是完整 JSON：

```json
{"choices":[{"delta":{"content":"这是"},"index":0}], ...}
```

`extractContent()` 使用 Jackson 解析 `choices[0].delta.content` 路径：

```java
private String extractContent(String data) {
    JsonNode root = objectMapper.readTree(data);
    JsonNode choices = root.path("choices");
    if (choices.isArray() && !choices.isEmpty()) {
        JsonNode delta = choices.get(0).path("delta");
        JsonNode content = delta.path("content");
        if (!content.isMissingNode()) {
            return content.asText();
        }
    }
    return "";
}
```

### 5.5 心跳机制

```java
private Flux<ServerSentEvent<String>> heartbeat() {
    return Flux.interval(Duration.ofSeconds(15))
        .map(i -> ServerSentEvent.<String>builder()
            .comment("keepalive")
            .build());
}
```

- 每 15 秒发送一个 SSE 注释 `:keepalive`
- 通过 `mergeWith()` 合并到主数据流中
- 客户端 EventSource 自动忽略注释行，不触发 `onmessage` 事件
- 防止 Nginx、Gateway 等中间层因静默而断开 TCP 连接

## 6. Circuit Breaker 熔断策略

### 6.1 Resilience4j 配置

**文件**：`ai-assistant-service/src/main/resources/application.yml`

```yaml
resilience4j:
  circuitbreaker:
    configs:
      default:
        sliding-window-size: 10                   # 统计最近 10 次调用
        sliding-window-type: COUNT_BASED          # 基于次数计数
        failure-rate-threshold: 50                # 失败率 >= 50% 触发熔断
        wait-duration-in-open-state: 30s          # 熔断后 30 秒尝试恢复
        permitted-number-of-calls-in-half-open-state: 3   # 半开状态允许 3 次探测
        automatic-transition-from-open-to-half-open-enabled: true
        slow-call-rate-threshold: 100             # 100% 慢调用视为失败
        slow-call-duration-threshold: 30s         # 超过 30s 为慢调用
    instances:
      openai-api:
        base-config: default
  timelimiter:
    instances:
      openai-api:
        timeout-duration: 120s                    # 整体超时 120 秒
```

### 6.2 状态机

```
                     ┌─────────────────────────────┐
                     │                             │
                     ▼                             │
              ┌──────────────┐                     │
         ┌───▶│   CLOSED     │                     │
         │    │  (正常调用)   │                     │
         │    └──────┬───────┘                     │
         │           │ 10 次调用中 >= 5 次失败      │
         │           ▼                             │
         │    ┌──────────────┐                     │
         │    │    OPEN      │                     │
         │    │  (直接降级)   │                     │
         │    └──────┬───────┘                     │
         │           │ 30 秒后                     │
         │           ▼                             │
         │    ┌──────────────┐                     │
         │    │  HALF_OPEN   │                     │
         │    │  (允许3次探测)│                     │
         │    └──┬───────┬───┘                     │
         │       │       │                         │
         │  探测成功   探测失败                      │
         │  (< 50%)   (>= 50%)                    │
         └──────────────┘                         │
```

### 6.3 降级策略

```java
private Flux<ServerSentEvent<String>> fallbackStream(Throwable throwable) {
    String msg;
    if (throwable != null && throwable.getMessage() != null
            && throwable.getMessage().contains("timeout")) {
        msg = "AI 服务响应超时，请稍后重试。";
    } else {
        msg = "AI 服务暂时不可用，请稍后再试。";
    }
    return Flux.just(ServerSentEvent.<String>builder()
            .data(msg)
            .build());
}
```

降级返回单条 SSE 事件，包含用户友好的错误提示，前端可据此展示 Toast 或提示信息。

## 7. Gateway 与 Nginx 路由配置

### 7.1 Gateway 路由

**文件**：`gateway-service/src/main/resources/application.yml`

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: ai-assistant-service
          uri: lb://ai-assistant-service
          predicates:
            - Path=/api/ai/**
          metadata:
            connect-timeout: 5000        # 连接超时 5 秒
            response-timeout: 300000     # 响应超时 300 秒 (5 分钟)
```

**关键参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| connect-timeout | 5000ms | 与服务建立连接的超时 |
| response-timeout | 300000ms (5 分钟) | SSE 为长连接，需足够长的超时以支持持续生成 |

### 7.2 Nginx 配置

**文件**：`config/nginx/nginx.conf`

```nginx
location /api/ai/ {
    proxy_pass http://gateway_cluster;

    # SSE 必须禁用代理缓冲 — 数据应立即推送给客户端
    proxy_buffering off;
    proxy_cache off;

    # 长连接超时
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;

    # 禁用响应压缩 — 流式数据逐块推送,压缩会等待完整响应
    gzip off;

    # 限流：AI 写接口，单 IP 突发 10 次请求/秒
    limit_req zone=write_limit burst=10 nodelay;
}
```

**为什么禁用缓冲和压缩**：

- **proxy_buffering off**：Nginx 默认会缓冲上游响应再发给客户端。对 SSE 流式数据，这会导致所有数据直到流结束才一次性发送，完全破坏了"打字机"效果
- **gzip off**：Gzip 压缩需要完整响应体才能压缩，同样会阻断流式传输
- **proxy_cache off**：缓存没有意义，每次 AI 生成的响应都不同

### 7.3 全链路超时协调

| 组件 | 超时配置 | 值 |
|------|---------|-----|
| 前端 fetch | AbortController signal | 用户可手动取消 |
| Nginx | proxy_read_timeout | 300s |
| Nginx | proxy_send_timeout | 300s |
| Gateway | response-timeout | 300s |
| AI Service | timeoutSeconds | 120s |
| LLM API | max_tokens | 2048 (自然结束) |

设计原则：**内层超时短于外层**。AI Service 的 120s 超时会先于 Gateway 的 300s 触发，用户收到明确的超时降级消息而非 TCP 连接被 Gateway 强行断开。

## 8. Demo 模式

### 8.1 启用条件

满足以下任一条件时自动进入 Demo 模式：

1. `AI_DEMO_MODE=true`（环境变量显式启用）
2. `OPENAI_API_KEY` 未配置或为空字符串

### 8.2 Demo 输出

8 句预设中文文本，按 2-4 字符分块，每块间隔 200ms：

```
"让我来为你展开分析这个问题。"
"从多个角度来看，这个主题有着丰富的内涵。"
"首先，我们需要理解其核心概念和基本原理。"
"深入探讨这个话题，我们会发现其中蕴含的深刻意义。"
"这不仅是一个理论问题，更与实践紧密相连。"
"综上所述，我们可以得出以下几点重要的认识。"
"在实际应用中，这个思路已经被广泛验证。"
"未来的发展方向值得我们持续关注和思考。"
```

**特点**：

- `Random(42)` 固定种子确保每次输出的分块方式一致，便于测试
- 末尾同样发送 `[DONE]` 结束标记
- 不通过熔断器，独立于真实 API 调用路径

### 8.3 Demo 模式的使用场景

| 场景 | 说明 |
|------|------|
| 本地开发与调试 | 无需申请 API Key 即可运行完整链路 |
| 前端开发联调 | 前端工程师无需配置任何后端服务 |
| 项目演示 | 稳定输出，不受 API 配额和网络影响 |
| CI/CD 集成测试 | 可复现的输出，适合自动化验证 SSE 解析逻辑 |

## 10. 未来改进方向

### 10.1 短期优化（阶段 7-8）

| 改进项 | 优先级 | 说明 |
|--------|--------|------|
| **Prompt 工程** | 高 | 当前 system prompt 为通用版，可针对不同文章类型（技术/文学/新闻）定制多套 prompt |
| **模型选择 API** | 中 | 在 `ContinueRequest` 中增加可选 `model` 字段，允许前端按场景选择模型 |
| **生成参数调优** | 中 | temperature 和 max_tokens 可按用户偏好配置，支持"创意模式"和"精确模式" |
| **Token 计数与限流** | 中 | 按用户统计单日 Token 消耗量，防止资源滥用 |

### 10.2 中期规划（阶段 9-10）

| 改进项 | 说明 |
|--------|------|
| **多轮对话支持** | 将历史消息附加到 messages 数组，实现连续对话续写 |
| **摘要/翻译/改写** | 扩展多 endpoint，各用不同 system prompt，复用同一 WebClient 基础设施 |
| **WebSocket 替代 SSE** | 对于需要双向通信的场景（如用户中途打断生成），WebSocket 更合适 |
| **流控与排队** | 使用 Redis + 令牌桶限制并发 LLM 调用数，超限请求排队等待 |

### 10.3 长期愿景

| 改进项 | 说明 |
|--------|------|
| **真实 LLM 替换 Demo** | 当前标签提取和合规检测均为规则模拟，应接入 LLM 做真正的语义理解 |
| **RAG 增强生成** | 结合向量数据库，让 AI 基于知识库内容续写，提升专业性和准确性 |
| **多模态** | 支持图片理解与生成（如为文章自动生成配图） |
| **Fine-tuning** | 基于平台优质内容微调模型，形成平台特有的写作风格 |
