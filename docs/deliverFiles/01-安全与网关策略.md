## 1. 系统安全架构总览

### 1.1 纵深防御体系

Knowledge Platform 采用多层纵深防御架构，安全控制分布在外部接入层、网关层、应用层和数据层四个维度：

```
                          INTERNET
                             |
                             v
                    +------------------+
                    |   Nginx :80      |  <-- 第1层：IP 限流 + 连接数限制
                    |   (反向代理)     |       gzip 压缩，前端 SPA 托管
                    +--------+---------+
                             |
                             v
                    +------------------+
                    |  Spring Cloud    |  <-- 第2层：JWT 认证 + 角色鉴权
                    |  Gateway :8080   |       Sentinel 应用级限流
                    |  (双实例)        |       CORS 跨域控制
                    +--------+---------+
                             |
              +--------------+--------------+
              |              |              |
              v              v              v
       +-----------+  +-----------+  +-----------+
       |  user-svc |  | article   |  | interact  |  <-- 第3层：BCrypt 密码
       |           |  | -svc      |  | -svc      |       所有权校验（admin/owner）
       +-----------+  +-----------+  +-----------+       @Transactional 事务
              |              |              |
              v              v              v
       +-----------+  +-----------+  +-----------+
       |  MySQL    |  |  Redis    |  | RabbitMQ  |  <-- 第4层：Druid SQL 防火墙
       |  (主+从)  |  |  (缓存)   |  | (消息队列)|       binlog 复制 + read-only 从库
       +-----------+  +-----------+  +-----------+       BCrypt 密码散列
```

### 1.2 安全组件全景

| 层次 | 组件 | 负责功能 | 关键文件 |
|------|------|----------|----------|
| Nginx | limit_req_zone / limit_conn | IP 级令牌桶限流，连接数控制 | `config/nginx/nginx.conf` |
| Gateway | JwtAuthFilter | JWT 解析、黑名单检查、角色校验 | `gateway-service/.../filter/JwtAuthFilter.java` |
| Gateway | Sentinel GatewayFilter | QPS 限流，Nacos 动态规则推送 | `gateway-service/.../config/SentinelRuleBootstrap.java` |
| Gateway | CorsConfig | 跨域访问控制 | `gateway-service/.../config/CorsConfig.java` |
| User Service | BCryptPasswordEncoder | 密码散列（登录+注册） | `user-service/.../service/impl/UserServiceImpl.java` |
| Article Service | checkOwnership() | 文章所有权校验（admin 旁路） | `article-service/.../service/impl/ArticleServiceImpl.java` |
| Common | ReadWriteRoutingDataSource | 读/写数据源分离 | `common/.../config/ReadWriteRoutingDataSource.java` |

| Compliance | ComplianceCheckConsumer | 敏感词审核、违规拦截 | `compliance-service/.../consumer/ComplianceCheckConsumer.java` |

## 2. JWT 身份认证体系

### 2.1 Token 生成

JWT 由 `JwtUtils` 工具类生成，使用 HMAC-SHA384 (HS384) 签名算法，依赖 jjwt 0.12.6 库。

**核心代码路径:** `common/src/main/java/com/kb/common/utils/JwtUtils.java`

```java
// 密钥派生（从静态常量转换为 HMAC 密钥）
private static final SecretKey KEY = Keys.hmacShaKeyFor(
        JwtConstants.SECRET.getBytes(StandardCharsets.UTF_8));

// Token 生成
public static String generateToken(Long userId, String role) {
    Date now = new Date();
    Date expiry = new Date(now.getTime() + JwtConstants.EXPIRATION_MS);
    return Jwts.builder()
            .subject(String.valueOf(userId))
            .claim("role", role)
            .issuedAt(now)
            .expiration(expiry)
            .signWith(KEY)
            .compact();
}
```

### 2.2 JWT 常量配置

**文件路径:** `common/src/main/java/com/kb/common/constant/JwtConstants.java`

| 常量 | 值 | 说明 |
|------|-----|------|
| `SECRET` | `System.getenv().getOrDefault("JWT_SECRET", "knowledge-platform-jwt-secret-key-2026-min-32-chars!!")` | HMAC-SHA384 签名密钥（环境变量注入，失败时回退默认值） |
|------|-----|------|

**Stage 13 修复后代码：**

```java
// After Stage 13 fix
public static final String SECRET = System.getenv().getOrDefault(
        "JWT_SECRET", "knowledge-platform-jwt-secret-key-2026-min-32-chars!!");
```

> **Stage 13 修复**: 密钥不再硬编码，改为优先读取环境变量 `JWT_SECRET`。生产环境可通过 K8s Secret 或 docker-compose `environment` 注入。默认值仅作为开发环境兜底。

| `EXPIRATION_MS` | `604,800,000` (7天) | Token 有效期 |
| `HEADER_NAME` | `"Authorization"` | HTTP 请求头名称 |
| `TOKEN_PREFIX` | `"Bearer "` | Token 前缀 |
| `HEADER_USER_ID` | `"X-User-Id"` | 下游传播的用户 ID 头 |
| `HEADER_USER_ROLE` | `"X-User-Role"` | 下游传播的角色头 |

### 2.3 Token 生命周期流程

```
  ┌──────────┐      ┌──────────┐      ┌───────────┐      ┌───────────┐
  │  Client  │      │  Nginx   │      │  Gateway  │      │ User-Svc  │
  └────┬─────┘      └────┬─────┘      └─────┬──────┘      └─────┬─────┘
       │                  │                  │                    │
       │  1. POST /api/user/login            │                    │
       │─────────────────────────────────────────────────────────>│
       │                  │                  │   2. BCrypt验证      │
       │                  │                  │   3. JwtUtils       │
       │                  │                  │     .generateToken()│
       │  4. {token, userId, username, role} │                    │
       │<─────────────────────────────────────────────────────────│
       │                  │                  │                    │
       │  5. GET /api/articles               │                    │
       │  Authorization: Bearer <token>      │                    │
       │─────────────────>──────────────────>│                    │
       │                  │                  │ 6. JwtAuthFilter   │
       │                  │                  │    解析 JWT        │
       │                  │                  │    检查黑名单       │
       │                  │                  │ 7. X-User-Id: 1    │
       │                  │                  │    X-User-Role:user│
       │                  │                  │───────────────────>│
       │                  │                  │                    │
       │  8. DELETE /api/user/logout          │                    │
       │  Authorization: Bearer <token>      │                    │
       │─────────────────>──────────────────>────────────────────>│
       │                  │                  │   9. token写入Redis │
       │                  │                  │      TTL=剩余有效期  │
       │                  │                  │                    │
```

**注销（Logout）机制:**

`UserServiceImpl.logout()` 将完整 JWT Token 写入 Redis 黑名单，Key 格式为 `token:blacklist:{token}`，TTL 等于该 Token 的剩余有效时间。网关 `JwtAuthFilter` 在每次请求时检查该 Key，命中则返回 401。

**文件路径:** `user-service/src/main/java/com/kb/user/service/impl/UserServiceImpl.java` (行 85-98)

```java
public void logout(String token) {
    Claims claims = JwtUtils.parseToken(token);
    long remainingTtl = JwtUtils.getRemainingTtl(claims);
    if (remainingTtl > 0) {
        String blacklistKey = RedisKeyConstants.TOKEN_BLACKLIST + token;
        redisTemplate.opsForValue().set(blacklistKey, "1", Duration.ofMillis(remainingTtl));
    }
}
```

## 3. 网关层安全策略

### 3.1 JwtAuthFilter -- 全局认证过滤器

**文件路径:** `gateway-service/src/main/java/com/kb/gateway/filter/JwtAuthFilter.java`

该过滤器实现 `GlobalFilter` 和 `Ordered` 接口，Order 设置为 `-100`（高优先级），对所有经过网关的请求执行以下流程：

```
                     请求进入
                         |
                         v
              +--------------------+
              | 1. pathMatcher     |
              |    .isPublicPath()?|──是──> 放行（无需认证）
              +--------------------+
                         | 否
                         v
              +--------------------+
              | 2. GET 方法       |
              |   && /api/articles |──是──> 放行（公开阅读）
              |   && /api/comments|
              |   (非 /mine)      |
              +--------------------+
                         | 否
                         v
              +--------------------+
              | 3. 提取            |
              |  Authorization头  |──缺失─> 401 "缺少或无效的认证令牌"
              +--------------------+
                         | 存在
                         v
              +--------------------+
              | 4. JwtUtils       |──异常─> 401 "令牌已过期/无效的令牌"
              |   .parseToken()   |
              +--------------------+
                         | 成功
                         v
              +--------------------+
              | 5. Redis 检查      |──命中─> 401 "令牌已注销"
              |   黑名单          |
              +--------------------+
                         | 未命中
                         v
              +--------------------+
              | 6. /admin/* 路由?  |──是 + 非admin角色─> 403 "需要管理员权限"
              +--------------------+
                         | 通过
                         v
              +--------------------+
              | 7. 传播 X-User-Id |
              |    X-User-Role    |──> 转发到下游服务
              +--------------------+
```

### 3.2 公开路径白名单

**文件路径:** `gateway-service/src/main/java/com/kb/gateway/filter/AuthPathMatcher.java`

| 分类 | 路径 | 说明 |
|------|------|------|
| 精确匹配 | `/api/user/register` | 用户注册 |
| 精确匹配 | `/api/user/login` | 用户登录 |
| 前缀匹配 | `/api/trending` | 热搜榜单及其子路径 |
| 精确匹配 | `/actuator/health` | 健康检查 |

### 3.3 GET 请求公开读取策略

为支持类似知乎的"无需登录即可浏览"体验，网关对以下 GET 请求跳过认证：

| 路径模式 | 例外 |
|----------|------|
| `/api/articles/**` | `/api/articles/mine` 仍需认证 |
| `/api/comments/**` | 无例外 |

这意味着：匿名用户可以浏览已发布文章列表、查看文章详情、查看评论；但查看"我的文章"、发表/编辑/删除文章、点赞/评论等写操作需要登录。

### 3.4 用户身份传播

认证成功后，JwtAuthFilter 将用户身份注入到下游请求头中，各微服务无需重复解析 JWT：

```java
// Stage 13 安全加固版本 — 先剥离客户端伪造头，再注入可信值
exchange.getRequest().mutate()
    .headers(h -> {
        h.remove(JwtConstants.HEADER_USER_ID);
        h.remove(JwtConstants.HEADER_USER_ROLE);
    })
    .header(JwtConstants.HEADER_USER_ID, String.valueOf(userId))
    .header(JwtConstants.HEADER_USER_ROLE, role)
    .build();
```

> **Stage 13 安全加固**: 网关在设置 X-User-Id 和 X-User-Role 头之前，先通过 `headers.remove()` 剥离客户端传入的同名头，防止请求头伪造攻击。此前客户端可直接传入 `X-User-Id` 和 `X-User-Role` 头来冒充任意用户。

下游服务通过读取这两个 Header 获取当前用户信息，无需再次解析 JWT Token。

### 3.5 CORS 跨域配置

**文件路径:** `gateway-service/src/main/java/com/kb/gateway/config/CorsConfig.java`

```java
// Stage 13 修复后
config.addAllowedOriginPattern("http://localhost:*");
config.addAllowedMethod("GET");
config.addAllowedMethod("POST");
config.addAllowedMethod("PUT");
config.addAllowedMethod("DELETE");
config.addAllowedMethod("OPTIONS");
config.addAllowedHeader("Content-Type");
config.addAllowedHeader("Authorization");
config.setAllowCredentials(true);
config.setMaxAge(3600L);
```

> **Stage 13 修复**: 已将 `allowedOriginPattern("*")` 改为 `localhost:*`，并明确列举 HTTP 方法和允许的请求头，消除 CORS 通配符 + credentials 组合漏洞。此前 `allowedOriginPattern("*")` 与 `allowCredentials(true)` 组合在规范上无效（浏览器会拒绝），且过度宽松的 CORS 策略存在跨站请求伪造风险。

## 4. 多层限流体系

系统采用 Nginx + Sentinel 双层限流，分别作用于不同维度：

### 4.1 第一层：Nginx IP 级限流

**文件路径:** `config/nginx/nginx.conf`

Nginx 使用 `ngx_http_limit_req_module` 和 `ngx_http_limit_conn_module` 实现令牌桶限流：

#### 4.1.1 限流规则总览

| 限流区域 | 速率 | 突发 | 适用路径 |
|----------|------|------|----------|
| `general_limit` | 100 req/s | 200 | 所有 `/api/` 路径（默认） |
| `write_limit` | 20 req/s | 5-10 | 注册、登录、发布、点赞、AI 续写 |
| `conn_limit` | 50 并发 | -- | 所有路径 |

#### 4.1.2 配置详情

```
# 限流规则定义
limit_req_zone $binary_remote_addr zone=general_limit:10m rate=100r/s;
limit_req_zone $binary_remote_addr zone=write_limit:10m   rate=20r/s;
limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

# 通用路由 — 100 req/s + burst 200
location /api/ {
    limit_req zone=general_limit burst=200 nodelay;
    proxy_pass http://gateway_cluster;
}

# 写接口 — 20 req/s + burst 5-10
location /api/user/register {
    limit_req zone=write_limit burst=5 nodelay;
    proxy_pass http://gateway_cluster;
}

location /api/user/login {
    limit_req zone=write_limit burst=10 nodelay;
    proxy_pass http://gateway_cluster;
}

location ~ ^/api/articles/(publish|like) {
    limit_req zone=write_limit burst=10 nodelay;
    proxy_pass http://gateway_cluster;
}

location /api/ai/ {
    limit_req zone=write_limit burst=10 nodelay;
    proxy_pass http://gateway_cluster;
}

# 热搜榜 — 放宽限流 + HTTP 缓存
location /api/trending {
    limit_req zone=general_limit burst=500 nodelay;
    proxy_cache trending_cache;
    proxy_cache_valid 200 1s;
    proxy_cache_lock on;
    proxy_pass http://gateway_cluster;
}
```

#### 4.1.3 热搜榜单三级缓存

```
 L0: Caffeine (应用内, 内存缓存, ~微秒级)
 L1: Redis (分布式缓存, ~毫秒级)
 L2: Nginx proxy_cache (边缘缓存, 1s TTL, cache_lock)
```

`proxy_cache_lock on` 确保在缓存过期瞬间，多个并发请求聚合为单次回源请求，避免缓存击穿。

### 4.2 第二层：Sentinel 应用级限流

**文件路径:** `gateway-service/src/main/java/com/kb/gateway/config/SentinelRuleBootstrap.java`

#### 4.2.1 规则管理架构

```
  +------------------+        gRPC 推送         +-----------------+
  |   Nacos          | -----------------------> |  Gateway        |
  |   config:        |        (实时生效)        |  SentinelRule   |
  |   gateway-       |                          |  Bootstrap      |
  |   sentinel-rules |                          |                 |
  +------------------+                          +--------+--------+
                                                        |
                                                        | 加载规则
                                                        v
                                               +--------+--------+
                                               |  GatewayRule    |
                                               |  Manager        |
                                               +--------+--------+
                                                        |
                                                        | 拦截请求
                                                        v
                                               +--------+--------+
                                               |  Sentinel       |
                                               |  GatewayFilter  |
                                               +-----------------+
```

#### 4.2.2 种子规则（classpath 兜底）

当 Nacos 不可用时，从 `classpath:sentinel/gateway-sentinel-rules.json` 加载种子规则作为兜底保护：

| 资源 | QPS 阈值 | 限流维度 | 说明 |
|------|----------|----------|------|
| user-service | 100 | 来源 IP (X-Real-IP) | 登录/注册等用户操作 |
| article-service | 50 | 来源 IP | 文章读写 |
| interact-service | 30 | 来源 IP | 评论/点赞 |

#### 4.2.3 动态规则更新

Sentinel 通过 Nacos gRPC Listener 自动接收规则变更，无需重启网关：

```java
configService.addListener(DATA_ID, GROUP, new Listener() {
    @Override
    public void receiveConfigInfo(String newConfig) {
        // 实时重新加载规则
        loadRules(newConfig, "Nacos");
    }
});
```

当 Nacos 配置被删除时，自动回退到种子规则，提供 fail-safe 保护。

#### 4.2.4 限流响应格式

**文件路径:** `gateway-service/src/main/java/com/kb/gateway/config/SentinelBlockHandlerConfig.java`

触发限流时返回结构化 JSON（HTTP 429 Too Many Requests），优于默认的纯文本 "Blocked by Sentinel"：

```json
{
    "code": 429,
    "message": "请求过于频繁，请稍后再试",
    "data": null
}
```

## 5. 权限与授权模型

### 5.1 角色定义

| 角色 | 数据库值 | 权限范围 |
|------|----------|----------|
| 普通用户 | `"user"` | 读写自己的文章、评论、点赞 |
| 管理员 | `"admin"` | 操作任意文章，访问 /admin/* 路由 |

### 5.2 网关层路由级鉴权

网关 `JwtAuthFilter` 对所有 `/admin/*` 路径执行角色检查：

```java
if (path.startsWith("/admin") && !"admin".equals(role)) {
    return writeError(exchange, ResultCode.FORBIDDEN, "需要管理员权限");
}
```

### 5.3 应用层资源级鉴权

**文件路径:** `article-service/src/main/java/com/kb/article/service/impl/ArticleServiceImpl.java` (行 184-191)

```java
private void checkOwnership(Article article, Long userId, String role) {
    if ("admin".equals(role)) {
        return;  // 管理员旁路 -- 可以操作任意的文章
    }
    if (!article.getAuthorId().equals(userId)) {
        throw new BusinessException(ResultCode.FORBIDDEN.getCode(), "无权操作此文章");
    }
}
```

该检查应用于以下操作：

| 操作 | 路由 | 调用方法 |
|------|------|----------|
| 发布文章 | `POST /api/articles/publish/{id}` | `publish()` |
| 更新文章 | `PUT /api/articles/{id}` | `update()` |
| 删除文章 | `DELETE /api/articles/{id}` | `delete()` |
| 我的文章列表 | `GET /api/articles/mine` | `listMine()` |

> **Stage 13 修复**: 文章创建时强制状态为 DRAFT，必须通过 `PUT /api/articles/{id}/publish` 发布。`publish()` 方法触发 RabbitMQ 消息发送，确保合规检测和标签提取对每篇已发布文章执行。

### 5.4 注册时角色自赋值风险

**文件路径:** `common/src/main/java/com/kb/common/dto/RegisterRequest.java`

```java
private String role;  // 客户端可自行传入 "admin"
```

**文件路径:** `user-service/src/main/java/com/kb/user/service/impl/UserServiceImpl.java` (行 48)

```java
user.setRole(request.getRole() != null && "admin".equals(request.getRole()) ? "admin" : "user");
```

> **安全风险:** 注册接口允许客户端传入 `"admin"` 角色，任何用户都可以注册为管理员。生产环境应移除 role 字段的用户输入，管理员账号应为后台手动创建或通过单独的审核流程授予。

## 6. 数据安全防护

### 6.1 密码存储 -- BCrypt

**文件路径:** `user-service/src/main/java/com/kb/user/service/impl/UserServiceImpl.java`

密码使用 Spring Security 的 `BCryptPasswordEncoder` 进行散列：

- 注册时：`passwordEncoder.encode(request.getPassword())`
- 登录时：`passwordEncoder.matches(request.getPassword(), user.getPassword())`
- 数据库 `users.password` 字段长度 200 字符（足够存储 BCrypt 60 字符散列值）

### 6.2 Druid SQL 防火墙

Druid 连接池启用 `wall` 过滤器，提供 SQL 注入防护：

```
filters: stat,wall
```

| 策略 | 配置 | 说明 |
|------|------|------|
| SQL 注入检测 | wall filter | 拦截常见 SQL 注入模式 |
| 多语句禁止 | `multi-statement-allow: false` | 防止 `; DROP TABLE` 攻击 |
| 慢 SQL 监控 | 1000ms 阈值 | 记录执行超过 1 秒的 SQL |
| 连接监控 | stat filter | 活跃连接、执行次数等指标 |

### 6.3 MySQL 主从复制安全

**文件路径:** `docker-compose.yml`

主库配置（server-id=1）：

```
--server-id=1
--log-bin=mysql-bin
--binlog-format=ROW
--binlog-do-db=kb_platform
```

从库配置（server-id=2）：

```
--server-id=2
--relay-log=relay-bin
--read-only=1    # 只读模式，防止误写
```

复制账户：`repl@'%'` 仅具备 `REPLICATION SLAVE` 和 `REPLICATION CLIENT` 权限。

**文件路径:** `config/sql/init-replication.sql`

### 6.4 读写分离 -- 路由逻辑

**文件路径:** `common/src/main/java/com/kb/common/config/ReadWriteRoutingDataSource.java`

```java
public class ReadWriteRoutingDataSource extends AbstractRoutingDataSource {
    public static final String MASTER = "MASTER";
    public static final String SLAVE = "SLAVE";

    @Override
    protected Object determineCurrentLookupKey() {
        if (TransactionSynchronizationManager.isCurrentTransactionReadOnly()) {
            return SLAVE;    // @Transactional(readOnly=true) -> 从库
        }
        return MASTER;       // @Transactional -> 主库
    }
}
```

路由规则：
- `@Transactional(readOnly = true)` --> 读从库
- `@Transactional` 或无事务 --> 写主库（默认）

## 7. 内容合规审核

### 7.1 审核架构

文章发布后，通过 RabbitMQ 消息队列触发异步合规检测：

```
  ArticleServiceImpl.publish()
           |
           | rabbitTemplate.convertAndSend("article.publish", ...)
           v
  +--------------------+         +---------------------------+
  | article.topic      | ------> | ComplianceCheckConsumer    |
  | .exchange          |         | (concurrency: 5-10)       |
  +--------------------+         +---------------------------+
           |                              |
           |                              | 1. 敏感词检测
           |                              | 2. 回写 audit_result
           v                              | 3. BLOCK -> 逻辑删除
  +--------------------+                  v
  | TagExtractConsumer |         +---------------------------+
  | (concurrency: 5)   |         | articles 表更新:          |
  | 标签提取 + 回写     |         | audit_result = PASS/     |
  +--------------------+         |               REVIEW/BLOCK|
                                 | deleted = 1 (BLOCK时)     |
                                 +---------------------------+
```

### 7.2 ComplianceCheckConsumer 审核逻辑

**文件路径:** `compliance-service/src/main/java/com/kb/compliance/consumer/ComplianceCheckConsumer.java`

| 审核结果 | 触发条件 | 系统行为 |
|----------|----------|----------|
| `PASS` | 无敏感词，内容长度 >= 10 | `audit_result = "PASS"` |
| `REVIEW` | 命中可疑词 OR 内容过短 | `audit_result = "REVIEW"`，需人工复核 |
| `BLOCK` | 命中敏感词 | `audit_result = "BLOCK"` + `deleted = 1`（逻辑删除） |

敏感词库（演示版）：
`"违规", "违法", "赌博", "色情", "暴力", "诈骗", "传销", "毒品", "枪支", "走私"`

可疑词库：
`"广告", "推广", "联系方式", "加微信", "QQ群", "免费领取", "点击链接", "代理", "返利"`

> **生产建议:** 当前为关键词匹配的简化实现，生产环境应接入专业内容审核服务（如阿里云内容安全、网易易盾），支持图片、视频等多模态检测，并考虑使用 LLM 进行语义级别的违规判断。

### 7.3 消息确认机制

消费者使用手动 ACK 模式（`ackMode = "MANUAL"`）：

- 处理成功：`channel.basicAck(deliveryTag, false)`
- 处理失败：`channel.basicNack(deliveryTag, false, false)`（不重新入队，避免死循环）

### 7.4 TagExtractConsumer 标签提取

**文件路径:** `tag-extract-service/src/main/java/com/kb/tag/consumer/TagExtractConsumer.java`

基于规则的标签提取逻辑：
1. 标题分词（按常见分隔符切分，去停用词）
2. 内容领域特征词匹配（15 个技术领域模式）
3. 合并去重，取前 5 个标签
4. 回写 `articles.tags` 字段

> **生产建议:** 当前为规则匹配实现，生产环境应接入 NLP 服务或 LLM API 进行语义标签提取。

  2. 管理员账号通过数据库直接创建或后台管理接口提升

#### SEC-004: 创建文章绕过合规审查（Mass Assignment）

```
Stage 13 修复前: 客户端可在 ArticleCreateRequest 中传入 status="PUBLISHED"，
  创建文章直接发布，绕过 RabbitMQ 合规审核流程

Stage 13 修复后:
  1. ArticleServiceImpl.create() 中强制 article.setStatus(ArticleStatus.DRAFT)
  2. 发布必须通过 PUT /api/articles/{id}/publish 接口
  3. publish() 方法触发 RabbitMQ 合规检测消息，确保每篇已发布文章经过审核

  1. JwtAuthFilter 先调用 headers.remove() 剥离客户端传入的同名头
  2. 再注入从 JWT 解析得到的可信 userId 和 role
  3. 下游服务读取的始终是经过 JWT 验证的身份信息
```
