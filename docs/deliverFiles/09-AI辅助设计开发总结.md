# AI 辅助设计开发总结

> 一个关于人机协作开发微服务知识平台的个人反思

---

## 1. 项目基本信息

- **项目名称**: Knowledge Platform — 基于 AI 的智能知识库与内容发布平台
- **技术栈**: Spring Cloud 微服务 (8 个服务) + React SPA 前端
- **代码规模**: ~8,000 行 Java + ~3,000 行前端代码
- **开发阶段**: 11 个（Stage 1-12，其中 Stage 2 为基础框架搭建，Stage 12 为集成测试）
- **开发周期**: 约 25 天
- **AI 工具**: Claude（Claude Code CLI + VSCode Extension）

### 项目架构概览

本项目是一个面向开发者的技术知识分享平台，核心功能包括用户注册登录、文章发布与管理、评论互动、点赞收藏、热搜榜单、以及基于 AI 的文章续写。系统采用微服务架构，包含以下服务：

| 服务 | 职责 |
|------|------|
| gateway-service | API 网关，统一鉴权、路由转发、Sentinel 限流 |
| user-service | 用户注册、登录、JWT 鉴权 |
| article-service | 文章 CRUD、搜索、热搜榜单 |
| interact-service | 评论、点赞、收藏 |
| ai-assistant-service | AI 流式续写（SSE）、Demo 模式 |
| notification-service | 异步消息广播（RabbitMQ） |
| common | 公共实体、工具类、统一返回格式 |

前端为纯静态 SPA（hash router），零构建工具，直接通过 Nginx 代理对接后端微服务。

## 2. 人机协作模式总结

### 2.1 各阶段 AI 参与度概览

整个开发过程按功能模块划分为 12 个阶段，每个阶段的 AI 参与方式与深度各不相同。下表汇总了各阶段的核心工作内容及 AI 的具体贡献。

| Stage | 内容 | 开发方式 | AI 代码占比 | AI 核心贡献 |
|-------|------|---------|------------|------------|
| 1-4 | 基础架构、用户、文章、互动 | AI 辅助 | ~80% | 生成 CRUD 代码、实体类、Controller、Service |
| 5 | 网关限流 | 自主完成 | ~30% | Sentinel 配置参考、规则设计建议 |
| 6 | 热搜榜单 | 自主完成 | ~30% | Redis ZSET 操作建议、热度计算公式设计 |
| 7 | 异步广播 | 自主完成 | ~40% | RabbitMQ 拓扑设计参考（Exchange/Queue/Binding） |
| 8 | AI 流式续写 | 自主完成 | ~20% | WebFlux SSE 代码框架、流式响应调试 |
| 9 | 高并发优化 | 自主完成 | ~30% | Nginx 反向代理配置、Caffeine 本地缓存集成 |
| 10 | 问题排查修复 | AI 辅助 | ~60% | Bug 诊断、日志分析、根因定位（AI 效率极高） |
| 11 | React 前端开发 | AI 生成 | ~90% | 全部前端代码（3 个 JS 模块 + CSS + HTML） |
| 12 | 集成测试 | AI 辅助 | ~70% | 48 个测试用例执行、缺陷记录、数据验证 |
| 13 | 代码审查与安全加固 | AI 辅助 | ~80% | 4 维审查（安全/功能/质量/运行时）、16 项修复、根因诊断 |
| 10 | 问题排查 | AI 辅助 | ~50% | Bug 诊断、日志分析、根因定位 |
| 11 | 前端开发 | AI 辅助 | ~90% | React 风格 SPA 组件、API 封装层、样式 |
| 12 | 集成测试 | AI 辅助 | ~70% | 测试用例执行、curl 脚本生成、缺陷记录 |

数据表明了一个清晰的规律：**越是模板化、重复性高的工作，AI 的代码占比越高；越是需要领域理解和架构权衡的工作，人的主导作用越突出。**

### 2.2 四种协作模式

回顾整个开发过程，我与 Claude 的协作可以归纳为以下四种典型模式：

#### 模式一：「AI 写，人审」

这是最高效的协作模式，也是整个项目中应用最广泛的模式。AI 负责生成代码的第一稿，人负责审查和修正。

**典型场景**：
- **Stage 11 前端开发**：AI 生成了完整的 index.html、CSS 样式、三个 JavaScript 模块（api.js / app.js / components.js），总计约 3,000 行代码。我只需定义整体架构（SPA hash router + 组件划分），AI 便自动完成了所有页面（登录、注册、文章列表、文章详情、文章编辑、AI 写作助手）的渲染和交互逻辑。
- **文档生成**：12 份开发日志和 10 份交付文档均由 AI 从代码和聊天记录反写，我仅做最终审核。

**核心经验**：这种模式下，人的价值在于「定义边界」——明确输入输出格式、异常处理策略、UI 交互规范，然后让 AI 在边界内自由发挥。审查时重点关注边界条件和安全问题即可。

#### 模式二：「人设计，AI 实现」

当涉及系统架构层面的决策时，由我完成顶层设计，AI 负责将设计转化为具体代码。

**典型场景**：
- **Stage 7 异步广播**：我先确定消息流转拓扑（Exchange 类型选择 Topic、Queue 命名规范、Binding Key 设计），AI 据此生成 RabbitMQ 配置类、消息生产者和消费者。
- **Stage 9 缓存策略**：我先设计了多层缓存架构（Caffeine 本地缓存 → Redis 分布式缓存 → MySQL 持久层），AI 完成具体的 Caffeine 配置类和缓存注解集成。

**核心经验**：这种模式下，人的价值在于「做减法」——AI 会给出多种实现方案，需要人根据业务场景（如数据一致性要求、延迟容忍度）做出取舍。如果直接让 AI 做架构决策，它往往会给出「大而全」但过度设计的方案。

#### 模式三：「人编码，AI 审查」

对于安全敏感和核心业务逻辑，我选择亲自编码，然后让 AI 进行代码审查。

**典型场景**：
- **网关 JwtAuthFilter**：JWT 解析、Token 验证、路径白名单匹配均由我手写，AI 审查发现了一个潜在的 Token 前缀处理漏洞。
- **Stage 5 Sentinel 限流规则**：限流阈值、降级策略由我根据业务场景设定，AI 检查配置的合理性和边界情况。

**核心经验**：这种模式下，AI 的价值在于「查漏补缺」——它能发现人容易忽略的边界条件（null 检查、并发安全、异常处理路径）。但安全相关的核心逻辑必须由人掌握，AI 审查是锦上添花而非雪中送炭。

#### 模式四：「AI 诊断，人修复」

这是 Bug 排查阶段最高效的协作模式。AI 负责分析日志、定位根因，人负责确认问题并实施修复。

**典型场景**：
- **Stage 10 问题排查**：AI 通过阅读网关日志、应用日志和配置文件，成功诊断了分页 total=0 问题（PageResult 未正确提取 MyBatis-Plus Page 对象的 total 字段）和 CORS 配置冲突问题。
- **Stage 12 集成测试**：AI 分析 curl 测试结果，识别出评论接口返回格式不一致、点赞去重逻辑缺失等问题。

**核心经验**：这种模式下，AI 相当于一个不知疲倦的「日志分析器」，能比人更快地扫描大量日志文件并定位异常。但最终修复方案仍需人的判断——AI 可能给出技术上正确但不符业合务预期的修复手段。

## 3. 核心场景的「人机协作迭代」记录

以下选取四个最具代表性的开发场景，详细记录人机协作的迭代过程。

### 3.1 场景一：网关 JwtAuthFilter 修复（Stage 12）

**背景**：在集成测试阶段发现，未登录用户可以访问 `/api/articles/mine` 接口并获取到「我的文章」列表，这明显存在鉴权绕过漏洞。

**第一轮 — AI 分析**：

我向 Claude 描述了问题现象后，它立即开始分析网关配置。Claude 读取了 `gateway-service/src/main/java/com/kb/gateway/filter/JwtAuthFilter.java`，迅速定位到关键代码：

```java
// 问题代码：匹配条件过宽
if (path.startsWith("/api/articles") && method == GET) {
    // 放行，不校验 Token
}
```

AI 指出：这段逻辑的本意是让未登录用户也可以浏览公开文章列表，但 `startsWith("/api/articles")` 的匹配范围太宽，导致 `/api/articles/mine`（需要登录才能查看的私有接口）也被放行了。

**第一轮 — 人的决策**：

我确认了 AI 的诊断，并制定了修复方案：在公开路径白名单中排除 `/api/articles/mine`，同时增加更精确的路径匹配逻辑，确保只有真正的公开浏览接口（如 `/api/articles/list`、`/api/articles/{id}`）可以通过 GET 方法不鉴权。具体的实现是在 `startsWith` 判断后增加一个排除条件，专门拦截 `mine` 路径。

**第二轮 — 修复与验证**：

AI 协助生成了修复后的代码，我审核后重新构建并部署了网关服务。然后 AI 编写了一组 curl 测试用例来验证修复效果：

- 未登录 GET `/api/articles/mine` → 401（修复前返回 200 + 数据，修复后正确拒绝）
- 未登录 GET `/api/articles/list` → 200（公共接口仍可正常访问）
- 已登录 GET `/api/articles/mine` → 200（鉴权通过后正常返回）

全部测试通过，问题成功修复。

**迭代总结**：2 轮即完成，AI 的分析阶段和验证阶段起到了关键作用，但修复方案的制定权始终在人的手中。这个场景体现了「AI 诊断 + 人决策 + AI 验证」的高效闭环。整个排查和修复过程耗时不到 30 分钟。

### 3.2 场景二：分页 total=0 Bug（Stage 10）

**背景**：前端调用文章列表接口时，返回的分页数据中 `total` 始终为 0，导致前端无法正确渲染分页组件（显示「共 0 条」但实际上有数据）。

**第一轮 — AI 排查**：

Claude 系统地检查了 article-service 的分页实现链路：
1. `ArticleController` 接收分页参数，调用 Service 层
2. `ArticleServiceImpl` 中创建 MyBatis-Plus 的 `Page` 对象并调用 `mapper.selectPage()`
3. 查询完成后，将 MyBatis-Plus `Page` 对象转换为自定义的 `PageResult` 返回

AI 快速锁定了问题所在——`PageResult` 的 `total` 字段未能从 MyBatis-Plus 的 `Page` 对象正确提取。在 `common/src/main/java/com/kb/common/vo/PageResult.java` 中，构造函数仅复制了 `records` 列表，遗漏了 `total` 字段的赋值。

**根因分析**：

这是一个典型的「胶水代码」问题——自定义的 `PageResult` 类与 MyBatis-Plus 的 `Page` 对象之间的转换逻辑不完整。更值得反思的是，这个 `PageResult` 类本身就是 AI 生成的（Stage 1 基础架构阶段），而后端分页逻辑也是 AI 生成的（Stage 3 文章模块），前端分页组件同样是 AI 生成的（Stage 11）。也就是说，AI 生成的代码组合在一起时，暴露了一个在独立审查中难以发现的缺陷。

**状态**：此问题被标记为低优先级 Bug，待后续版本修复。因为在集成测试环境中，前端临时使用了硬编码的分页逻辑作为 workaround——虽然用户体验不够完美，但不影响核心功能的验证。

**反思**：这个 Bug 暴露了一个重要现象：**AI 生成的代码在独立审查时可能没有问题，但代码之间的「契约」——接口约定、数据格式转换、异常传播——如果不在同一轮对话中生成，就容易出现不一致。** 这提示了一个改进方向：在涉及跨模块的数据传递时，应该先由人明确定义契约（如 PageResult 的完整字段列表），再让 AI 分别在上下游实现。

### 3.3 场景三：AI SSE Demo 模式验证（Stage 12）

**背景**：AI 辅助写作功能在 Demo 模式下运行，无需真实的外部 AI API Key。集成测试阶段需要验证 Demo 模式的 SSE（Server-Sent Events）流式输出是否正常工作，以及输出内容是否可用于前端开发和调试。

**AI 的贡献**：

Claude 编写了一套完整的 SSE 测试方案：

```bash
# 发起 SSE 请求并实时查看流式输出
curl -N -H "Authorization: Bearer $TOKEN" \
  "http://localhost:8080/api/ai/continue?content=Spring%20Boot"
```

AI 还编写了 awk 脚本来解析原始 SSE 数据流，统计 `data:` 块的数量、验证每个块的格式、计算块之间的时间间隔。这些分析帮助确认了流式输出的稳定性和性能特征。

**测试结果**：
- SSE 连接成功建立，响应头 `Content-Type: text/event-stream` 正确
- 共收到 51 个 `data:` 块，每个块包含一段约 20-50 字的 Demo 续写文本
- 块与块之间的间隔稳定在 80-120ms，模拟了真实 AI 的流式输出节奏
- 流在整个传输过程中无中断、无乱码、无重复

**关键发现**：

这个验证过程揭示了一个重要的开发策略：**在 AI 功能开发中，Demo/Mock 模式的设计投入是值得的，甚至是必要的。** Demo 模式带来了三个关键收益：

1. **降低开发门槛**：无需申请和配置外部 AI 服务的 API Key，新开发者克隆代码后即可运行完整功能。
2. **使自动化测试成为可能**：Demo 输出是确定性的（同样的输入总是产生同样的 51 个块），因此可以编写精确的断言来验证 SSE 行为。
3. **隔离外部依赖**：集成测试不依赖外部 AI 服务的可用性，避免了「外部 API 挂了导致 CI 全红」的尴尬。

这个发现将影响我未来所有涉及外部 API 集成的项目设计——永远为外部依赖提供一个可靠的 Demo/Mock 模式。

### 3.4 场景四：React 前端全 AI 生成（Stage 11）

这是整个项目中 AI 代码占比最高的阶段（~90%），也是最能体现「人机协作」模式价值的场景。这个场景值得详细拆解，因为它展示了当前 AI 辅助前端开发的真实能力上限和下限。

**人的工作（约 1 小时）**：

在让 AI 写任何一行代码之前，我先完成了以下设计工作：
1. **架构决策**：确定使用纯静态 SPA（单页应用），hash router（`#/login`、`#/articles` 等），零构建工具（不引入 webpack/vite，直接使用 ES6 模块），通过 Nginx 反向代理解决跨域问题。
2. **模块划分**：`api.js`（API 调用封装，统一的 fetch 封装 + JWT Token 自动附加 + 错误处理）、`app.js`（SPA 路由调度 + 全局状态管理）、`components.js`（所有 UI 组件的渲染函数和事件绑定）。
3. **页面结构定义**：登录页、注册页、文章列表页、文章详情页、文章编辑页、AI 写作助手页，共 6 个页面。每个页面定义了核心交互流程。
4. **样式风格指定**：现代简约风格，蓝白配色，响应式布局（桌面端 1200px+、平板端 768-1199px、手机端 < 768px），卡片式文章列表。

**AI 的工作（约 30 分钟生成 + 1 小时微调）**：

在明确了以上设计后，我将任务拆解为四个连续的 prompt 交给 Claude：

1. **生成 index.html**：完整的 HTML 骨架，包含顶部导航栏、主内容区容器、模态框容器、以及所有页面的基础 HTML 模板。AI 一次性生成了约 200 行结构良好的 HTML。

2. **生成 css/style.css**：约 800 行 CSS，包含 CSS 变量定义（颜色、间距、圆角）、全局重置样式、导航栏样式、卡片样式、表单样式、按钮样式、模态框样式、响应式断点样式。AI 生成的 CSS 保持了良好的一致性——所有颜色通过变量引用而非硬编码。

3. **生成 js/api.js**：约 300 行 JavaScript，封装了所有后端 API 调用。每个 API 函数返回 Promise，自动附加 Authorization Header，统一的错误处理（401 自动跳转登录页、网络错误友好提示）。代码结构清晰，每个 API 端点独立一个函数。

4. **生成 js/app.js + js/components.js**：约 2,000 行 JavaScript，实现了 SPA 路由（监听 `hashchange` 事件）、全局状态管理（用户登录态、文章数据缓存）、以及所有 6 个页面的渲染和交互逻辑。每个组件函数遵循一致的签名：`renderXxxPage(container, data)` + `bindXxxEvents()`。

**成果评估**：

代码质量方面：
- 可读性良好，函数命名规范，注释适量
- 无明显 ESLint 级别的错误（未使用变量、隐式全局变量等）
- 但存在一些代码重复（如多个组件中重复的 fetch 调用模式），可以通过提取公共函数优化

功能完整性方面：
- 所有计划的 6 个页面均实现并可用
- 核心交互流程（登录→查看文章→点赞→评论）端到端可用
- SSE 流式输出在 AI 写作助手中正确渲染

已知缺陷：
- 部分 API 调用的错误处理不够细致（网络超时、服务不可用、业务错误未区分处理）
- 分页组件在边界情况下（总页数为 1、当前页为最后一页时）有 UI 显示缺陷
- 移动端响应式适配不够完善，部分按钮在窄屏上的触碰区域过小

**核心感悟**：

前端是 AI 辅助开发的「甜蜜点」。原因有三：
1. UI 代码高度模板化——一个列表页的代码结构和另一个列表页 80% 相似
2. 前端逻辑可预测——用户点击 → 发请求 → 渲染结果 → 绑定新事件
3. 视觉反馈即时——代码对不对，刷新页面就能看到

在这个场景中，人的价值体现在「定义整体的信息架构和交互规范」——哪些数据放在哪个页面、用户的操作路径是什么、错误状态如何呈现。而 AI 在「将设计转化为代码」这一步展现了极高的效率。

如果说传统前端开发中 80% 的时间花在写代码、20% 花在设计，那么 AI 辅助下这个比例可以反转为 20% 设计、80% 审查和微调。对于我来说，这是一种更有满足感的开发方式——我可以花更多时间思考「这个页面应该长什么样」，而不是「怎么用 CSS 把这个按钮居中」。

### 3.5 场景五：消费者"静默失败"根因诊断（Stage 13）

**背景**：集成测试发现标签提取和合规检测始终不生效 — `tags` 和 `auditResult` 字段始终为 NULL。AI 生成了消费者代码，但代码从未实际执行成功。

**诊断过程（AI 主导）**：

1. AI 首先检查 Docker 容器日志，发现 tag-extract-service 和 compliance-service 启动正常，RabbitMQ 连接建立成功，但无"收到任务"日志
2. AI 对比生产者（article-service）和消费者（tag-extract-service）的配置差异
3. 关键发现：article-service 配置了 `Jackson2JsonMessageConverter` Bean（消息以 JSON 序列化并设置 `content-type: application/json`），而两个消费者服务完全没有配置任何 `MessageConverter`
4. AI 解释了根因：Spring AMQP 的默认 `SimpleMessageConverter` 遇到 `application/json` 类型时返回 `byte[]`，无法将 `byte[]` 转换为 `@RabbitListener` 方法的 `Map<String, Object>` 参数，消息处理失败但被 MANUAL ACK 模式静默忽略
5. AI 生成修复代码：两个消费者各添加 `RabbitMQConfig` 配置类

**验证**：重新部署后，消费者立即开始处理积压消息：
- `收到标签提取任务: articleId=28 → 标签提取完成: tags=草稿测试`
- `收到合规检测任务: articleId=28 → auditResult=REVIEW`

**人的决策**：确认修复方案、重新编译部署、验证端到端流程。

**迭代总结**：这是整个项目中最有价值的 AI 诊断之一。从"功能不工作"到"根因定位"，AI 通过跨服务配置对比在几分钟内找到了问题，如果人工排查可能需要 1-2 小时。这个场景完美体现了「AI 诊断，人修复」协作模式的优势——AI 在模式匹配和跨文件对比方面远超人类，而人负责决策和执行。

## 4. 关键经验与反思

### 4.1 AI 擅长的领域

经过 25 天的高强度协作，我对 Claude 的能力边界有了更清晰的认识。以下是 AI 在这个项目中表现最出色的五个领域：

**1. 模板化代码生成**

Controller、Service、Mapper 层的 CRUD 代码质量极高。对于标准化的数据库操作（增删改查、分页查询、条件筛选），AI 生成的代码基本无需修改即可使用。在 Stage 1-4 中，AI 一次性生成了约 2,000 行的样板代码，覆盖了用户、文章、评论、点赞四个模块的全部 CRUD 接口。这些代码遵循了一致的命名规范（`getXxxById`、`listXxxByCondition`、`createXxx`、`updateXxx`、`deleteXxx`），异常处理模式统一，日志记录规范。

**2. 配置文件编写**

Nginx 反向代理配置、Docker Compose 多服务编排、Spring Boot application.yml——这些结构化配置是 AI 的天然强项。AI 不仅生成速度极快，而且能在不同配置文件之间保持一致性（如端口号、服务名、环境变量不会出现拼写不一致）。在 Stage 9 的高并发优化中，AI 生成的 Nginx 配置（upstream 负载均衡、静态资源缓存策略、gzip 压缩）只需微调超时参数即可投入使用。

**3. 文档生成**

从代码反写文档是 AI 的「杀手级应用」。开发日志、API 文档、部署指南——这些文档的质量/速度比（quality per unit time）远超人工编写，估计达到 5 倍以上。尤其是在项目后期，AI 维护的开发日志成了重要的知识资产。举个例子：在 Stage 12 集成测试时遇到一个问题，我通过搜索 Stage 7 的开发日志找到了当时 RabbitMQ 配置的设计决策记录，5 分钟就理解了当初的设计意图——如果没有日志，可能需要重新阅读代码半小时。

**4. 测试执行与结果解析**

批量 curl 测试 + JSON 响应解析 + 结构化报告输出——AI 在这条链路上的表现令人印象深刻。Stage 12 中，AI 编写并执行了 50+ 条测试用例，并自动将结果分类为「通过」「失败（预期行为不符）」和「失败（服务错误）」，最终生成了一份格式规整的测试报告。这极大提升了测试效率：如果人工执行同样的 50 条测试并整理报告，预计需要 3-4 小时，而 AI 在 1 小时内完成。

**5. Bug 诊断**

AI 读取日志并定位根因的能力超出我的预期。在 Stage 10 的问题排查中，AI 在 10 分钟内完成了 3 个 Bug 的根因分析（分页 total=0、CORS 配置冲突、评论接口返回格式不一致），而如果由我来做，可能需要 1-2 小时。关键优势在于 AI 的「全局视角」——它能同时关联网关日志、应用日志和配置文件，而人往往需要来回切换上下文，容易遗漏线索。

### 4.2 AI 的局限性

认识 AI 的局限性与认识其优势同等重要。以下四种场景中，人的主导作用不可替代：

**1. 架构决策**

AI 可以给出架构建议，但无法理解业务做出合理的取舍。例如，在缓存策略设计中，AI 建议使用 Redis 作为唯一缓存层以保证数据一致性。但我根据实际需求（热点文章访问 QPS 高但数据变更频率低），选择了 Caffeine 本地缓存 + Redis 分布式缓存的两层架构。这个决策基于对业务特征的理解——「热点文章的读取延迟需要做到毫秒级，而数据一致性可以接受秒级延迟」——这些上下文信息是 AI 无法自主获取的。AI 可以告诉你「技术上可以做 A 或 B」，但无法告诉你「在你的业务场景下 A 比 B 更合适」。

**2. 安全敏感代码**

JWT Token 验证、CORS 配置、SQL 注入防护——这些安全相关的代码必须由人审查和确认。AI 生成的 JwtAuthFilter 初版存在 Token 前缀验证缺失的问题（没有检查 Token 是否以 "Bearer " 开头就直接截取），如果不经审查直接使用，可能导致严重的安全漏洞。安全领域的原则是「信任但要验证」，对于 AI 生成的安全代码，验证的标准应该比人工代码更高——因为 AI 缺乏对安全威胁模型的真正理解，它只是基于训练数据中的安全模式生成代码。

**3. 性能调优**

AI 可以识别明显的性能反模式（如 N+1 查询、未使用连接池），但无法深入理解某个特定系统的性能瓶颈。在 Stage 9 的高并发优化中，我通过 JMeter 压测定位到两个核心瓶颈：数据库连接池大小不足（默认 10 连接在高并发下成为瓶颈）和 Redis 默认的 JDK 序列化方式性能低下。AI 在之前的代码生成中并未指出这些问题，因为它们不是「代码层面的错误」而是「运行时的配置不当」。性能调优需要人对系统运行时行为的深入理解，以及压测工具的熟练使用——这些是 AI 目前难以替代的。

**4. 复杂业务规则**

热度计算公式（浏览量权重 + 点赞权重 + 评论权重 + 时间衰减因子）、评论审核规则——这些业务规则的精确形式必须由人定义。举个例子，AI 初始建议的热度公式是简单的线性加权，但在实际测试中发现，线性加权导致新文章永远无法进入热榜（旧文章的累积数据优势太大）。我根据对内容社区的理解，增加了时间衰减因子（发布超过 7 天的文章热度按指数衰减），这个调整使热榜的时效性和多样性显著改善。AI 可以提供初始方案，但规则的微调（如衰减系数从 0.95 调整到 0.92 对榜单稳定性的影响）需要人对业务的直觉和反复实验。

### 4.3 五条最佳实践

基于本次项目经验，我总结了五条可复用的 AI 辅助开发最佳实践：

**1. 先让 AI 理解上下文**

在开始任何开发任务前，提供充分的上下文信息——`CLAUDE.md` 项目说明、架构图、API 文档、数据库 ER 图。上下文越丰富，AI 生成的代码与项目现有代码风格的一致性越高。本项目在 `CLAUDE.md` 中维护了项目结构说明、技术栈版本、编码规范等信息，显著提升了 AI 对新增代码的命名风格和模块划分的准确度。一个反例是：在没有更新 `CLAUDE.md` 的情况下，让 AI 新增一个服务模块，结果它使用了与其他服务不同的包命名规范（`com.kb.xxx` vs `com.kb.service.xxx`），导致后续集成时需要批量重命名。

**2. 分阶段迭代，每阶段人确认**

不要一次性给 AI 一个大型任务期望它全部完成。将其拆分为小阶段，每个阶段完成后由人审查确认。本项目 Stage 1-12 的划分正是遵循了这个原则——每个阶段有明确的验收标准（如 Stage 3 的验收标准是「文章 CRUD 全部通过 curl 测试」），AI 的产出在进入下一阶段前必须通过人的审查。这避免了「AI 生成 5000 行代码后发现有架构级错误需要全部重写」的风险。同时，分阶段迭代也让人始终对项目有掌控感——你知道每一行代码的作用和来源。

**3. AI 写测试 + 人审查 > 人写测试**

在测试覆盖面上，AI 明显优于人。人可以想到 5 个测试场景，AI 可以生成 20 个，包括那些「理论上不应该发生」的边界条件（如传入负数页码、传入超长标题、传入特殊字符）。但人需要审查测试用例的合理性——AI 有时会生成逻辑上矛盾或用例重复的测试。最佳实践是：AI 生成测试框架和用例矩阵，人审查补充关键业务场景（如「用户点赞自己的文章」这种产品逻辑上需要特殊处理的场景），AI 执行测试并输出结构化报告。

**4. 让 AI 维护开发日志**

这是本次项目最高 ROI（投资回报率）的实践。在 `开发日志/` 目录下维护了 stage1 到 stage12 的详细记录，每篇约 500-800 字，由 AI 在每阶段结束时根据聊天记录自动生成。这些日志在后期成为宝贵的参考文档——当 Stage 12 集成测试遇到问题时，我可以通过搜索日志快速回溯「这个功能是什么时候加的？当初的设计意图是什么？」。如果由人来维护，考虑到写完代码已疲惫不堪，大概率会省略或简化。

**5. 关键决策始终由人拍板**

这个原则在项目的每一个阶段都得到了贯彻。AI 提供选项、分析利弊，但最终的选择权——无论是技术选型（Sentinel vs Resilience4j）、架构决策（同步调用 vs 异步消息）、还是 UI 设计（列表页 vs 卡片页）——都由我做出。AI 是参谋，不是决策者。它可以穷举可能的方案，但无法为方案的后果负责——因为「后果」这个概念本身就不在 AI 的理解范围内。当 AI 建议「使用 Kafka 替代 RabbitMQ 以获得更高的吞吐量」时，我需要权衡：团队对 RabbitMQ 的熟悉程度、运维复杂度、当前业务的吞吐量需求是否真的需要 Kafka。这些权衡中的人的因素（团队技能、运维成本）是 AI 无法充分评估的。

### 4.4 效率数据

以下是对本次项目 AI 辅助效果的量化评估：

| 指标 | 数据 |
|------|------|
| 整体开发时间节省 | 估计 40-50% |
| AI 生成代码占比（总体） | ~70% |
| 最高 AI 代码占比阶段 | Stage 11 前端（~90%） |
| 最低 AI 代码占比阶段 | Stage 8 AI 流式续写（~20%） |
| AI 发现 Bug 数量 | ~15 个 |
| AI 生成文档数量 | 10 份交付文档 + 12 份开发日志 |
| 人均代码审查轮次 | 约 2-3 轮/阶段 |

一个值得注意的数据点：**AI 代码占比与任务的「创新性」呈负相关**。当任务涉及全新的技术领域（如 Stage 8 的 WebFlux SSE 流式响应——此前项目中从未使用过响应式编程），或者需要深度集成的技术（如 Stage 7 的 RabbitMQ 消息拓扑——需要理解消息的完整生命周期），AI 的代码占比显著下降。这说明当前阶段的 AI 更适合「重新组合已知模式」而非「创造新模式」。

另一个有趣的观察：**AI 代码占比最高（Stage 11 前端 ~90%）并不意味着人的工作量最少**。恰恰相反，Stage 11 是我投入精力最多的阶段之一——因为审查 3,000 行 AI 生成的 JavaScript 代码并不比手写轻松多少。区别在于工作性质：审查和微调比从零编写更不消耗「创造性能量」，我可以保持更高的专注度和判断力。

## 5. 对未来项目的建议

基于本次项目的人机协作经验，我为未来的类似项目提出以下五条建议：

### 5.1 项目启动时即配置 AI 友好的开发环境

在项目初始化的第一天，就建立 `CLAUDE.md` 项目说明文件，包含：
- 项目目标与业务背景
- 技术栈与版本号（精确到版本，如 Spring Boot 2.7.x、MyBatis-Plus 3.5.x）
- 目录结构与模块划分（及每个模块的职责说明）
- 编码规范与命名约定（包名、类名、方法名的命名模式）
- 关键架构决策记录（ADR，记录每个重大技术决策的原因和权衡）

这个投入只需 30 分钟，但在后续 25 天的开发中持续产生回报。AI 对项目上下文的理解深度直接决定了代码生成的准确度和一致性。

### 5.2 建立 AI 代码审查 CI 流程

将 `/code-review` 集成到 CI/CD 流水线中。每次提交代码后，自动触发 AI 代码审查，重点关注：
- 潜在的空指针异常（Java 中未做 null 检查的 Optional 使用）
- 资源泄露（未关闭的数据库连接、IO 流）
- 并发安全问题（共享可变状态未加锁）
- SQL 注入风险（MyBatis 中使用 `${}` 而非 `#{}`）
- 硬编码的敏感信息（密码、API Key、Token）

AI 审查不能替代人工审查，但可以在人工审查之前过滤掉 80% 的低级错误，让人专注于架构和业务逻辑层面的审查。这种「两层审查」模式（AI 过滤低级错误 → 人审查架构和业务逻辑）预计可以提升整体审查效率 30-50%。

### 5.3 复杂任务使用多 Agent 并行编排

在 Stage 11 前端开发中，如果采用多 Agent 并行模式（一个 Agent 负责 HTML/CSS，一个负责 API 封装，一个负责组件逻辑），预计可以再节省 30% 的时间。Claude Code 支持 Worktree 隔离的并行 Agent，适合这种可拆解的大任务。

具体做法：
- 将大任务拆解为 3-5 个独立子任务，每个子任务有明确的输入（API 文档、设计稿）和输出（代码文件）
- 每个子任务分配给一个 Agent，在独立的 Worktree 中工作
- 由一个「协调者」Agent 负责合并代码并检查模块间的一致性

这种方式对任务的拆解粒度要求很高——拆得太粗，Agent 之间会产生冲突；拆得太细，协调成本超过并行收益。需要根据项目特点找到最佳粒度。

### 5.4 将 AI 视为「超级实习生」

这是我在此次项目中对 AI 角色最准确的定位：
- **实习生**：速度快，不知疲倦，能处理大量机械性工作；但需要明确指导，产出需要审查，不能独立承担责任。会犯低级错误，但不会犯同一个错误两次（在同一个会话中）。
- **超级**：知识面广（远超任何初级工程师），能生成高质量的第一稿，能在短时间内给出多种方案供选择。不会遗忘，不会遗漏细节（只要被提醒过），始终保持一致的代码风格。

这个心理模型有助于建立正确的协作心态：不要期望 AI 一次就完美，但要相信它能极大加速「从 0 到 1」的过程。把 AI 的产出当作「高级初稿」，而不是「最终交付物」。就像你不会让实习生独立负责生产环境的部署一样，你也不应该让 AI 未经审查的代码直接上线。

### 5.5 维护清晰的文档供 AI 理解上下文

这一点反复出现是有原因的——它是所有 AI 辅助开发效率的前提。具体建议：
- **API 文档**：维护 OpenAPI/Swagger 规范，AI 可以直接读取并生成客户端代码
- **数据模型**：维护实体关系图和表结构说明，AI 可以据此生成正确的 DTO/VO
- **架构图**：用文字描述或 Mermaid 图表记录服务间的调用关系和数据流
- **变更日志**：每次架构变更记录在 `CLAUDE.md` 的 ADR 部分

文档不需要长篇大论，但需要保持更新。过时的文档比没有文档更危险——它会误导 AI 生成与当前架构不兼容的代码。一个简单的维护策略是：在每次 PR 合并时，检查是否需要更新 `CLAUDE.md`，如果需要则作为 PR 的一部分一起提交。

## 6. 结语

25 天，8 个微服务，1 个前端应用，约 11,000 行代码。如果纯人工开发，我估计需要 35-40 天。AI 在整个过程中节省了约 40-50% 的时间，但更重要的是，它改变了我的开发方式。

最深刻的体会是：**AI 不是替代开发者，而是放大了开发者的能力**。它让我可以专注于「做什么」和「为什么做」，而将「怎么做」的大量细节委托出去。这种分工不是简单的任务外包，而是一种需要不断磨合的协作关系——我知道 AI 的优势和盲区，AI 通过我提供的上下文和我对其输出的反馈逐步适应项目的风格。

具体来说，我的角色在这 25 天中发生了一个微妙但深刻的转变：从「编码者」变成了「设计者 + 审查者」。过去我会把 70% 的精力花在写代码上、30% 花在设计上。现在这个比例反转为 30% 设计、50% 审查、20% 编码（核心逻辑）。这种转变让我感到更加充实——设计和审查比纯粹的编码更接近「创造」的本质。

这种协作模式对开发者的素质要求不是降低了，而是转移了。过去，一个好的开发者需要写得一手好代码；现在，一个好的 AI 辅助开发者需要：
- **清晰的系统设计能力**：定义 AI 的工作边界，将模糊的需求转化为精确的指令
- **精准的需求表达能力**：用自然语言准确描述期望的输出格式、边界条件、异常处理策略
- **快速的代码审查能力**：在大量 AI 生成的代码中快速识别逻辑错误、安全漏洞、代码异味
- **果断的技术决策能力**：在 AI 提供的多个方案中做选择，并对选择负责

这些能力不是天生的，而是在 25 天的实践中逐步磨练出来的。回想 Stage 1 时，我对 AI 的产出几乎全部接受——结果发现很多代码虽然在语法上正确，但在项目中并不协调（命名风格不一致、异常处理模式不统一、过度设计简单场景）。到了 Stage 12，我已经能在一个 prompt 中准确地定义边界条件、指定异常处理策略、甚至预判 AI 可能会犯什么错误并预先在 prompt 中纠正。这种对 AI 行为的「直觉」本身，就是从反复的实践中积累的宝贵经验。

另一个重要的成长是学会了「何时信任 AI，何时怀疑 AI」。当 AI 说「这个问题很简单」时，我学会了追问「你确定吗？检查一下边界情况」。当 AI 给出一个看起来太完美的方案时，我学会了用「如果 X 失败了会发生什么」这样的问题来探测方案的鲁棒性。这种批判性思维不是 AI 教我的，而是通过与 AI 的反复博弈中自己体会出来的。

让我用一个具体的例子来说明这种成长。在项目的早期，我给 AI 的 prompt 是这样的：

> 「帮我写一个文章的 CRUD 接口」

到项目后期，我的 prompt 变成了这样：

> 「帮我写一个文章的 CRUD 接口。要求：1) 使用 MyBatis-Plus 的 BaseMapper，不要手写 SQL；2) 分页接口使用 IPage，每页默认 10 条；3) 创建和更新接口需要校验标题（1-100 字符）和内容（至少 10 字符）；4) 删除接口是软删除；5) 所有接口统一使用 com.kb.common.vo.Result 作为返回类型；6) 异常使用 GlobalExceptionHandler 统一处理，不要在 Controller 里写 try-catch；7) 方法命名遵循 getXxx / listXxx / createXxx / updateXxx / deleteXxx 的规范。」

这两个 prompt 的区别，就是我在 25 天中最大的成长。我学会了如何「精确地表达需求」——这是 AI 辅助开发时代最核心的技能。

回到项目本身。Knowledge Platform 从一个想法变成了一个可以实际运行的系统——11,000 行代码，8 个微服务，所有核心功能可用，集成测试通过。如果没有 AI 的辅助，我不确定是否能在 25 天内完成。但我同样不确定的是：如果完全依赖 AI 而没有人来把握方向、做决策、审查代码，这个项目最终会变成什么样子。

或许这就是「人机协作」的最佳状态：人负责方向和决策，AI 负责执行和加速。两者各司其职，互相补位。人在这个过程中不仅没有「被替代」，反而因为有了 AI 的辅助，可以去思考更高层次的问题、尝试更大胆的设计、探索更广阔的可能性。

---

**最后一个建议给正在阅读本文的你**：不要等到「准备好」才开始使用 AI 辅助开发。最好的学习方式就是实践——找一个真实的项目，从第一天就使用 AI，在犯错中学习，在迭代中进步。AI 辅助开发的能力将成为未来软件开发者的核心素养，越早开始积累经验，越能在技术浪潮中占据主动。毕竟，我们正处在一个「AI 不会取代开发者，但会用 AI 的开发者会取代不会用的」的时代。
