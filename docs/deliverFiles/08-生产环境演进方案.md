## 1. 当前状态评估

### 1.1 部署架构概览

当前系统基于 **Docker Compose** 单机部署，所有服务运行在同一台宿主机上。总计 12 个容器实例，其中 11 个为单实例运行，仅 Gateway 部署了 2 个实例用于基础的负载均衡。

```
                          Internet
                             |
                         [Port 80]
                             |
                     +-------v--------+
                     |  kb-nginx      |  nginx:alpine
                     |  反向代理 + 限流 |  静态资源 Gzip + HTTP 缓存
                     +-------+--------+
                             | (least_conn -> gateway_cluster)
                     +-------+--------+
                     |                |
              +------v------+  +-----v-------+
              | kb-gateway-1|  | kb-gateway-2|  Spring Cloud Gateway
              | :8080        |  | :8080        |  + Sentinel 限流 + Nacos 发现
              +------+-------+  +-----+--------+
                     |                |
              +------v----------------v--------+
              |        Nacos 2.2 Standalone    |  注册/配置中心 x1
              +-------------------------------+
                     |                |
        +------------+-----------------+------------+
        |            |                 |            |
   [kb-user-svc]  [kb-article-svc] [kb-interact] [kb-ai-assistant]
   用户服务         文章服务           互动服务       AI 流式服务
        |            |                 |
   +----v----+  +---v---+     +-------v--------+
   | MySQL 8 |  | Redis |     | RabbitMQ 3     |
   | (主+从)  |  | 7.x   |     | management      |
   +---------+  +-------+     +----------------+

  [kb-tag-extract]    [kb-compliance]
   AI 标签提取消费者     AI 合规检测消费者
   (RabbitMQ Consumer)  (RabbitMQ Consumer)
```

### 1.3 现有架构优势

- **Dockerfile 已遵循部分最佳实践**: 所有 Java 服务使用 `eclipse-temurin:17-jdk-alpine` 基础镜像，均已创建非 root 用户 `appuser` 并以 `USER appuser` 运行。
- **前端 multi-stage build**: `frontend-react/Dockerfile` 使用 Node 18 Alpine -> Nginx Alpine 两阶段构建，最终镜像仅含静态资源，体积约 30MB。
- **健康检查已有基础**: MySQL (`mysqladmin ping`)、Redis (`redis-cli ping`)、RabbitMQ (`rabbitmq-diagnostics check_port_connectivity`)、Nacos (HTTP readiness) 均配置了 healthcheck，且服务间 `depends_on` 使用了 `condition: service_healthy`。
- **读写分离已部署但未激活**: MySQL 主从部署完成 (server-id 1/2, binlog + relay-log)，应用层配置了 `DB_HOST` 和 `DB_SLAVE_HOST` 环境变量，但代码层读写分离路由逻辑待落地。
- **Nginx 限流与缓存成熟**: 已实现令牌桶限流 (general 100r/s, write 20r/s), 连接数限制 (50/IP), HTTP 缓存 (trending 1s TTL, `proxy_cache_lock` 合并并发请求), SSE 长连接支持 (300s timeout, `proxy_buffering off`)。
- **Sentinel 网关限流**: Gateway 已集成 `spring-cloud-alibaba-sentinel-gateway` + `sentinel-datasource-nacos`, eager 模式加载规则，支持 Dashboard 可视化。

### 1.4 当前风险与不足

| 风险类别 | 具体问题 | 影响等级 |
|----------|----------|----------|
| 单点故障 | 除 Gateway 外所有服务单实例，任意服务挂掉即不可用 | **高** |
| 数据库 | MySQL 主从已部署但无自动故障转移，从库读流量未路由 | **高** |
| 缓存 | Redis 单实例，仅 RDB 持久化 (默认配置)，宕机可能丢失最多 15 分钟数据 | **高** |
| 消息队列 | RabbitMQ 单实例，无镜像队列，宕机后内存中消息丢失 | **中** |
| 注册中心 | Nacos standalone 模式，宕机后新服务无法注册/发现 | **高** |
| 可观测性 | 无监控系统 (Prometheus/Grafana)、无集中日志 (ELK/Loki)、无链路追踪 | **高** |
| 告警 | 服务宕机无通知，完全依赖人工发现 | **高** |
| 安全 | JWT Secret、DB 密码硬编码在 docker-compose.yml 中，OPENAI_API_KEY 以环境变量传入 | **中** |
| 持久化 | Docker volume 本地存储，无异地备份 | **中** |
| 资源隔离 | 无 Docker `mem_limit` / `cpus` 约束，容器可能争抢宿主机资源 | **低** |

### 1.5 当前网络架构

```
kb-network (bridge driver)
  + -- kb-nginx      -> 公网入口 :80
  + -- kb-gateway-1  -> :8080 (内部)
  + -- kb-gateway-2  -> :8080 (内部)
  + -- kb-nacos      -> :8848 (内部)
  + -- kb-mysql      -> :3306 (内部) + :3306 (宿主机)
  + -- kb-mysql-slave-> :3306 (内部) + :3307 (宿主机)
  + -- kb-redis      -> :6379 (内部) + :6379 (宿主机)
  + -- kb-rabbitmq   -> :5672, :15672 (内部 + 宿主机)
  + -- [所有 Java 服务] -> 无端口暴露 (仅内部通信)
```

Nginx 上游配置 (`nginx.conf`):

```nginx
upstream gateway_cluster {
    least_conn;
    server gateway-1:8080 weight=1 max_fails=3 fail_timeout=30s;
    server gateway-2:8080 weight=1 max_fails=3 fail_timeout=30s;
    keepalive 64;
}
```

## 2. Phase 1 -- 高可用加固 (1-2 周)

> **目标**: 在现有 Docker Compose 基础上消除所有单点故障，建立基础的故障恢复能力。
> **原则**: 不改变部署方式 (仍使用 Docker Compose)，不引入新基础设施，仅做配置和服务层面的加固。

### 2.1 MySQL: 主从自动故障转移

**当前问题**: 主从已部署但主库故障时无自动切换，从库读流量未路由。

**实施方案**:

选用 **Orchestrator** (轻量级，GitHub 7k+ stars，适合中小规模) 而非 MHA (已停止维护):

```yaml
# docker-compose.yml 新增 mysql-orchestrator 服务
mysql-orchestrator:
  image: openarkcode/orchestrator:latest
  container_name: kb-orchestrator
  ports:
    - "3000:3000"
  volumes:
    - ./config/orchestrator/orchestrator.conf.json:/etc/orchestrator.conf.json:ro
  environment:
    - ORCHESTRATOR_API=:"3000"
    - ORCHESTRATOR_DEBUG=true
  networks:
    - kb-network
  depends_on:
    mysql:
      condition: service_healthy
    mysql-slave:
      condition: service_healthy
```

**Orchestrator 关键配置** (`config/orchestrator/orchestrator.conf.json`):

```json
{
  "Debug": true,
  "ListenAddress": ":3000",
  "MySQLTopologyUser": "orc_user",
  "MySQLTopologyPassword": "orc_pass",
  "MySQLOrchestratorHost": "mysql-orchestrator",
  "MySQLOrchestratorPort": 3306,
  "DiscoverByShowSlaveHosts": true,
  "InstancePollSeconds": 5,
  "RecoveryPeriodBlockSeconds": 3600,
  "RecoverMasterClusterFilters": ["kb_platform"],
  "RecoverIntermediateMasterClusterFilters": ["*"],
  "ApplyMySQLPromotionAfterMasterFailover": true,
  "FailMasterPromotionOnLagMinutes": 1,
  "PostMasterFailoverProcesses": [
    "echo 'Master failover completed' | wall"
  ]
}
```

**备份策略** (宿主机 crontab):

```bash
# 每日 02:00 全量备份
0 2 * * * docker exec kb-mysql mysqldump \
  --single-transaction --quick --lock-tables=false \
  -uroot -proot123 kb_platform \
  | gzip > /backup/mysql/full-$(date +\%Y\%m\%d).sql.gz

# 保留 30 天全量 + 7 天 binlog
find /backup/mysql/ -name "full-*.sql.gz" -mtime +30 -delete
```

**读写分离落地**: 在各服务代码层通过 AbstractRoutingDataSource 或 ShardingSphere-JDBC 实现读写分离路由，使 `DB_SLAVE_HOST` 和 `DB_SLAVE_PORT` 环境变量真正生效。读操作 (SELECT) 自动路由到从库，写操作路由到主库。

### 2.2 Redis: Sentinel 哨兵模式

**当前问题**: 单实例，宕机后缓存全量丢失，仅默认 RDB 持久化。

**实施方案**:

部署 1 主 2 从 + 3 Sentinel (共 6 个容器):

```yaml
# Redis 主节点
redis-master:
  image: redis:7-alpine
  container_name: kb-redis-master
  command: >
    redis-server
    --requirepass ${REDIS_PASS:-redis123}
    --appendonly yes
    --appendfsync everysec
    --aof-use-rdb-preamble yes
    --save 900 1
    --save 300 10
    --save 60 10000
  volumes:
    - redis_master_data:/data
  networks:
    - kb-network
  healthcheck:
    test: ["CMD", "redis-cli", "-a", "${REDIS_PASS:-redis123}", "ping"]
    interval: 5s
    timeout: 3s
    retries: 3

# Redis 从节点 x2
redis-replica-1:
  image: redis:7-alpine
  container_name: kb-redis-replica-1
  command: >
    redis-server
    --replicaof kb-redis-master 6379
    --masterauth ${REDIS_PASS:-redis123}
    --requirepass ${REDIS_PASS:-redis123}
    --appendonly yes
    --appendfsync everysec
  volumes:
    - redis_replica1_data:/data
  networks:
    - kb-network
  depends_on:
    redis-master:
      condition: service_healthy

redis-replica-2:
  image: redis:7-alpine
  container_name: kb-redis-replica-2
  command: >
    redis-server
    --replicaof kb-redis-master 6379
    --masterauth ${REDIS_PASS:-redis123}
    --requirepass ${REDIS_PASS:-redis123}
    --appendonly yes
    --appendfsync everysec
  volumes:
    - redis_replica2_data:/data
  networks:
    - kb-network
  depends_on:
    redis-master:
      condition: service_healthy

# Sentinel x3 (奇数节点，避免脑裂)
redis-sentinel-1:
  image: redis:7-alpine
  container_name: kb-redis-sentinel-1
  command: redis-sentinel /etc/redis/sentinel.conf
  volumes:
    - ./config/redis/sentinel-1.conf:/etc/redis/sentinel.conf:ro
  networks:
    - kb-network
  depends_on:
    - redis-master
    - redis-replica-1
    - redis-replica-2

# sentinel-2, sentinel-3 类似
```

**Sentinel 配置** (`config/redis/sentinel-1.conf`):

```
port 26379
sentinel monitor kb-redis-master kb-redis-master 6379 2
sentinel down-after-milliseconds kb-redis-master 3000
sentinel failover-timeout kb-redis-master 10000
sentinel parallel-syncs kb-redis-master 1
sentinel auth-pass kb-redis-master redis123
```

**Spring Boot 客户端变更**: 各服务 `application.yml` 改用 Sentinel 模式连接:

```yaml
spring:
  redis:
    sentinel:
      master: kb-redis-master
      nodes:
        - redis-sentinel-1:26379
        - redis-sentinel-2:26379
        - redis-sentinel-3:26379
    password: ${REDIS_PASS:redis123}
```

### 2.3 RabbitMQ: 镜像队列

**当前问题**: 单实例，宕机后内存中未消费的消息永久丢失。

**实施方案**:

部署 3 节点 RabbitMQ 集群 + 镜像队列策略:

```yaml
rabbitmq-1:
  image: rabbitmq:3-management-alpine
  container_name: kb-rabbitmq-1
  hostname: rabbitmq-1
  environment:
    - RABBITMQ_ERLANG_COOKIE=${RABBITMQ_COOKIE:-secret-cookie}
    - RABBITMQ_DEFAULT_USER=${RABBITMQ_USER:-guest}
    - RABBITMQ_DEFAULT_PASS=${RABBITMQ_PASS:-guest}
  volumes:
    - rabbitmq1_data:/var/lib/rabbitmq
    - ./config/rabbitmq/rabbitmq.conf:/etc/rabbitmq/rabbitmq.conf:ro
  networks:
    - kb-network
  healthcheck:
    test: ["CMD", "rabbitmq-diagnostics", "check_port_connectivity"]
    interval: 10s
    timeout: 5s
    retries: 5

rabbitmq-2:
  image: rabbitmq:3-management-alpine
  container_name: kb-rabbitmq-2
  hostname: rabbitmq-2
  environment:
    - RABBITMQ_ERLANG_COOKIE=${RABBITMQ_COOKIE:-secret-cookie}
  volumes:
    - rabbitmq2_data:/var/lib/rabbitmq
  networks:
    - kb-network
  # 启动后执行: rabbitmqctl join_cluster rabbit@rabbitmq-1

rabbitmq-3:
  # ... 同 rabbitmq-2
```

**镜像队列策略** (集群启动后执行):

```bash
# 进入任一节点
docker exec kb-rabbitmq-1 rabbitmqctl set_policy ha-mirror \
  "^(article\.tag\.|article\.compliance\.|article\..*\.)" \
  '{"ha-mode":"exactly","ha-params":2,"ha-sync-mode":"automatic"}'
```

- `ha-mode=exactly, ha-params=2`: 每条消息在 2 个节点上保留镜像 (1 主 + 1 镜像)
- `ha-sync-mode=automatic`: 新加入节点自动同步数据

**Spring Boot 客户端变更**:

```yaml
spring:
  rabbitmq:
    addresses: rabbitmq-1:5672,rabbitmq-2:5672,rabbitmq-3:5672
    username: ${RABBITMQ_USER:guest}
    password: ${RABBITMQ_PASS:guest}
```

### 2.4 Nacos: 集群模式

**当前问题**: standalone 模式，宕机后注册中心不可用。

**实施方案**: 部署 3 节点 Nacos 集群，使用已有 MySQL 主库存储配置数据:

```yaml
nacos-1:
  image: nacos/nacos-server:v2.2.0
  container_name: kb-nacos-1
  environment:
    - MODE=cluster
    - NACOS_SERVERS=nacos-1:8848 nacos-2:8848 nacos-3:8848
    - NACOS_APPLICATION_PORT=8848
    - SPRING_DATASOURCE_PLATFORM=mysql
    - MYSQL_SERVICE_HOST=mysql
    - MYSQL_SERVICE_PORT=3306
    - MYSQL_SERVICE_DB_NAME=nacos_config
    - MYSQL_SERVICE_USER=root
    - MYSQL_SERVICE_PASSWORD=root123
    - NACOS_AUTH_ENABLE=true
    - NACOS_AUTH_TOKEN=SecretKey012345678901234567890123456789012345678901234567890123456789
  volumes:
    - nacos1_data:/home/nacos/data
  networks:
    - kb-network

nacos-2:
  # ... NACOS_SERVERS 相同，端口设为 8848 (Docker 内部)

nacos-3:
  # ... 同上
```

**Nginx 代理 Nacos 集群** (在 `nginx.conf` 中追加):

```nginx
upstream nacos_cluster {
    server nacos-1:8848 weight=1;
    server nacos-2:8848 weight=1;
    server nacos-3:8848 weight=1;
}

# 修改原有 /nacos/ 路由
location /nacos/ {
    proxy_pass http://nacos_cluster/nacos/;
}
```

**客户端变更**:

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-1:8848,nacos-2:8848,nacos-3:8848
```

### 2.5 服务多实例扩展

移除 `container_name` 固定名称，改用 `docker-compose --scale`:

```bash
# 将关键服务扩展到 2 实例
docker-compose up -d --scale user-service=2 \
  --scale article-service=2 \
  --scale interact-service=2 \
  --scale tag-extract-service=2 \
  --scale compliance-service=2 \
  --scale ai-assistant-service=2
```

> **注意**: `container_name` 与 `--scale` 冲突，需从各服务的 Compose 定义中移除 `container_name` 行。

### 2.7 日志集中

**统一 JSON 格式** (logstash-logback-encoder):

```xml
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

**Logback 配置** (`logback-spring.xml`):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <includeMdcKeyName>traceId</includeMdcKeyName>
            <includeMdcKeyName>spanId</includeMdcKeyName>
            <includeMdcKeyName>userId</includeMdcKeyName>
            <customFields>{"service":"${SERVICE_NAME:-unknown}"}</customFields>
        </encoder>
    </appender>

    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>/var/log/app/app.log</file>
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <includeMdcKeyName>traceId</includeMdcKeyName>
        </encoder>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>/var/log/app/app.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>7</maxHistory>
        </rollingPolicy>
    </appender>

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
    </root>
</configuration>
```

**日志卷挂载** (每个服务 `docker-compose.yml` 添加):

```yaml
article-service:
  volumes:
    - ./logs/article-service:/var/log/app
```

### 3.1 容器化完善 -- Dockerfile 优化

当前 Dockerfile 已有 non-root 用户，但缺少 multi-stage 优化 (Java 服务) 和容器感知 JVM 参数。优化后:

```dockerfile
# ============================================================
# Stage 1: Maven 编译 (所有 Java 服务统一)
# ============================================================
FROM maven:3.9-eclipse-temurin-17-alpine AS builder
WORKDIR /build
# 先复制依赖文件，利用 Docker layer cache
COPY pom.xml .
COPY common/pom.xml common/
RUN mvn dependency:go-offline -B -q
# 复制源码并编译
COPY . .
RUN mvn package -DskipTests -B -q && \
    java -Djarmode=layertools -jar target/*.jar extract \
      --destination target/extracted

# ============================================================
# Stage 2: 运行镜像 (最小化)
# ============================================================
FROM eclipse-temurin:17-jre-alpine
LABEL maintainer="knowledge-platform"
LABEL service="${SERVICE_NAME}"

# 安全: 非 root 用户
RUN addgroup -S -g 1000 appgroup && \
    adduser -S -u 1000 appuser -G appgroup
RUN mkdir -p /app /var/log/app && \
    chown -R appuser:appgroup /app /var/log/app

WORKDIR /app

# 利用 Spring Boot layertools 分层复制 (提升缓存利用率)
COPY --from=builder --chown=appuser:appgroup \
  /build/target/extracted/dependencies/ ./
COPY --from=builder --chown=appuser:appgroup \
  /build/target/extracted/spring-boot-loader/ ./
COPY --from=builder --chown=appuser:appgroup \
  /build/target/extracted/snapshot-dependencies/ ./
COPY --from=builder --chown=appuser:appgroup \
  /build/target/extracted/application/ ./

EXPOSE 8080 8081

USER appuser

# 容器感知 JVM 参数
ENV JAVA_OPTS="-XX:+UseContainerSupport \
  -XX:MaxRAMPercentage=75.0 \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200"

ENTRYPOINT ["sh", "-c", "java ${JAVA_OPTS} \
    -Djava.security.egd=file:/dev/./urandom \
    org.springframework.boot.loader.launch.JarLauncher"]
```

**优化对比**:

| 优化项 | 之前 | 之后 |
|--------|------|------|
| 基础镜像 | `jdk-alpine` (含 JDK) | `jre-alpine` (仅 JRE) |
| 分层 | 整 JAR 复制 | layertools 分层 (依赖层缓存) |
| 内存配置 | 硬编码 Xms/Xmx | `MaxRAMPercentage` (容器感知) |

### 4.4 链路追踪 (TraceId)

Gateway 层生成 TraceId 并通过 HTTP Header 传播:

```java
// Gateway Filter: 生成或透传 TraceId
@Bean
public GlobalFilter traceIdFilter() {
    return (exchange, chain) -> {
        String traceId = exchange.getRequest().getHeaders()
            .getOrDefault("X-Trace-Id",
                List.of(UUID.randomUUID().toString()))
            .get(0);
        exchange.getResponse().getHeaders().add("X-Trace-Id", traceId);
        return chain.filter(exchange)
            .contextWrite(ctx -> ctx.put("traceId", traceId));
    };
}
```

各服务从 Request Header 提取 TraceId 并注入 MDC:

```java
@Component
public class TraceInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request,
            HttpServletResponse response, Object handler) {
        String traceId = request.getHeader("X-Trace-Id");
        if (traceId == null) traceId = UUID.randomUUID().toString();
        MDC.put("traceId", traceId);
        return true;
    }
}
```

通过 `logstash-logback-encoder` 的 `<includeMdcKeyName>traceId</includeMdcKeyName>` 将 traceId 输出到每条日志。

### 4.5 Alerting: Grafana AlertManager

#### 告警分级

| 级别 | 名称 | 条件 | 持续 | 通知方式 |
|------|------|------|------|----------|
| **P0** | 服务宕机 | `up{job!=""} == 0` | 1min | 电话 + 钉钉 |
| **P0** | 错误率飙升 | `error_rate > 5%` | 2min | 电话 + 钉钉 |
| **P1** | P99 延迟过高 | `P99_latency > 2000ms` | 5min | 钉钉群 @全体 |
| **P1** | DB 连接池耗尽 | `hikaricp_active / hikaricp_max > 0.9` | 3min | 钉钉群 |
| **P1** | RabbitMQ 消息积压 | `rabbitmq_queue_messages_ready > 1000` | 5min | 钉钉群 |
| **P1** | Redis 内存满 | `redis_memory_used / redis_memory_max > 0.85` | 5min | 钉钉群 |
| **P2** | 磁盘使用过高 | `disk_available / disk_total < 0.2` | 10min | 钉钉群 |
| **P2** | 内存使用过高 | `mem_available / mem_total < 0.1` | 10min | 钉钉群 |
| **P2** | 证书即将过期 | `cert_expiry_days < 30` | 24h | 邮件 |

### 9.3 故障恢复 SOP

#### 场景 1: MySQL 主库宕机

```
1. [告警触发] P0: MySQL 主库心跳丢失 >1min
2. [确认] 运维登录 Orchestrator Dashboard 确认主库状态
3. [决策] 主库宕机 >2min -> 触发故障转移 (Orchestrator 或手动)
4. [执行] 从库提升为新主库
5. [验证] curl /api/trending 返回 200
6. [恢复] 启动新从库实例，从新主库同步
7. [通报] 发送故障报告到钉钉群
8. [复盘] 事后分析根因，更新 Runbook
```

#### 场景 2: 全集群不可用 (K8s 控制面故障)

```
1. [触发] 所有服务 503
2. [确认] kubectl get nodes 检查节点状态
3. [恢复]
   a. 单节点故障: 驱逐 Pod，K8s 自动重调度
   b. etcd 损坏: 从 Velero 恢复 etcd 快照
   c. 全集群: Velero 恢复 -> 重建集群 -> 恢复 PV
4. [切流] DNS 切换至灾备集群 (如果有)
5. [验证] 冒烟测试通过
6. [复盘] 事后分析
```
