# 05-实时通信设计

> **文档版本**: 1.0  
> **最后更新**: 2026-07-06  
> **适用分支**: stage6+  
> **相关服务**: ai-assistant-service, article-service, interact-service  

---

## 1. 概述

Knowledge Platform 中存在两种"实时"通信机制，分别解决不同层面的问题：

| 机制 | 用途 | 通信模式 | 底层技术 |
|------|------|---------|---------|
| **SSE (Server-Sent Events)** | AI 流式续写，将 LLM 生成 token 逐字推送至浏览器 | 客户端-服务器单向流 | Spring WebFlux + Reactor Flux\<ServerSentEvent\> |
| **Redis Pub/Sub** | 跨实例缓存失效，通知所有 article-service 实例刷新本地 Caffeine 缓存 | 服务器间广播 | Redis Pub/Sub + Caffeine TTL |

两者互补：SSE 解决"如何把数据实时推送给用户"，Pub/Sub 解决"多实例间如何即时同步状态"。

## 2. SSE 实时流式续写

### 2.1 业务场景

用户在前端编辑器中输入文章上文，点击"AI 续写"按钮，AI 服务调用 OpenAI 兼容 API，将生成的文本逐 token 以 SSE 格式推送至浏览器，实现"打字机"效果。

### 2.2 接口定义

```
POST /api/ai/continue
Content-Type: application/json
Accept: text/event-stream

Request Body: { "context": "人工智能是当今科技领域..." }

Response:
Content-Type: text/event-stream

data: 最

data: 具

data: 变革

data: 性的

data: 技术之一。

data: [DONE]
```

**Controller 层** — `AiContinueController.java`：

```java
// 文件: ai-assistant-service/src/main/java/com/kb/ai/controller/AiContinueController.java

@PostMapping(value = "/api/ai/continue", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> continueStream(
        @Valid @RequestBody ContinueRequest request,
        @RequestHeader(value = JwtConstants.HEADER_USER_ID, defaultValue = "0") Long userId) {

    return aiStreamService.continueStream(request.getContext(), userId)
            .doOnComplete(() -> log.info("AI 续写完成: userId={}", userId))
            .doOnCancel(() -> log.info("客户端断开连接: userId={}", userId))
            .doOnError(e -> log.error("AI 续写异常: userId={}, error={}", userId, e.getMessage()));
}
```

关键设计点：
- **返回类型 `Flux<ServerSentEvent<String>>`**：Spring WebFlux 自动将 Flux 序列化为 SSE 流，每个 ServerSentEvent 对应一个 `data:` 行。
- **`doOnCancel`**：客户端断开（关闭浏览器标签页、网络中断）时触发，用于释放上游 LLM 连接资源。
- **`doOnError`**：LLM API 调用失败、超时、熔断降级时触发，记录日志但不影响已推送的内容。

### 2.3 SSE 全链路架构

```
Browser EventSource
    │
    │  HTTP/1.1, Accept: text/event-stream
    ▼
┌─────────────────────────────────────────────┐
│  Nginx (反向代理层)                           │
│  - proxy_buffering off    ← SSE 必须禁用缓冲   │
│  - proxy_read_timeout 300s ← 长连接超时        │
│  - proxy_send_timeout 300s                    │
│  - gzip off              ← 流式数据逐块推送     │
│  - limit_req zone=write_limit burst=10       │
└──────────────┬──────────────────────────────┘
               │
               │  least_conn → gateway_cluster
               ▼
┌─────────────────────────────────────────────┐
│  Spring Cloud Gateway (网关层)                │
│  - route: /api/ai/** → ai-assistant-service  │
│  - metadata.response-timeout: 300000 (5min)  │
│  - metadata.connect-timeout: 5000            │
│  - Reactive, 非阻塞                            │
└──────────────┬──────────────────────────────┘
               │
               │  lb://ai-assistant-service (Nacos)
               ▼
┌─────────────────────────────────────────────┐
│  ai-assistant-service (WebFlux Reactor)      │
│  - @PostMapping produces text/event-stream   │
│  - WebClient → OpenAI API (stream=true)      │
│  - Circuit Breaker (Resilience4j)            │
│  - Demo mode fallback                        │
└─────────────────────────────────────────────┘
```

#### Nginx 层配置

```nginx
# 文件: config/nginx/nginx.conf

location /api/ai/ {
    proxy_pass http://gateway_cluster;

    # SSE 必须禁用代理缓冲 — 否则数据会积压在 nginx 缓冲区，
    # 直到连接关闭才一次性推送给客户端
    proxy_buffering off;
    proxy_cache off;

    # 长连接超时 — AI 续写最长 5 分钟
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;

    # 禁用响应压缩 — 流式数据需要逐块推送，压缩会等待完整响应
    gzip off;

    # 限流 — 写接口，单 IP 最多 10 burst
    limit_req zone=write_limit burst=10 nodelay;
}
```

#### Gateway 层配置

```yaml
# 文件: gateway-service/src/main/resources/application.yml

spring:
  cloud:
    gateway:
      routes:
        - id: ai-assistant-service
          uri: lb://ai-assistant-service
          predicates:
            - Path=/api/ai/**
          metadata:
            connect-timeout: 5000
            response-timeout: 300000   # 5分钟，匹配 Nginx proxy_read_timeout
```

### 2.4 心跳设计

**问题**：Nginx 默认 `proxy_read_timeout` 为 60 秒。如果 LLM 生成暂停超过 60 秒（例如用户只发了一句很短的上文，LLM 很快生成完毕但前端还在等待），代理会断开连接。即使显式配置了 300 秒，某些中间网络设备（如 AWS ALB、企业防火墙）也有默认的空闲超时。

**解决方案**：每 15 秒发送一条 SSE 注释作为心跳。

```java
// 文件: ai-assistant-service/src/main/java/com/kb/ai/service/impl/OpenAiStreamServiceImpl.java

/**
 * 每 15 秒发送一次 SSE 注释作为心跳，防止网关/代理超时断开.
 *
 * SSE 注释以冒号开头，浏览器 EventSource 会忽略，
 * 但 TCP 连接保持活跃.
 */
private Flux<ServerSentEvent<String>> heartbeat() {
    return Flux.interval(Duration.ofSeconds(15))
            .map(i -> ServerSentEvent.<String>builder()
                    .comment("keepalive")
                    .build());
}
```

心跳流通过 `mergeWith(heartbeat())` 与正常数据流合并：

```java
return webClient.post()
        .uri("/v1/chat/completions")
        // ... 构建请求、解析响应 ...
        .mergeWith(heartbeat())        // ← 合并心跳
        .timeout(Duration.ofSeconds(aiConfig.getTimeoutSeconds()))
        .doOnCancel(() -> log.debug("上游 LLM 请求取消"));
```

`mergeWith` 确保即使 LLM 长时间无输出，TCP 连接每 15 秒也有数据流动，防止中间层超时断开。

### 2.5 超时与异常处理

| 超时类型 | 配置位置 | 默认值 | 作用 |
|---------|---------|-------|------|
| 连接建立超时 | `ChannelOption.CONNECT_TIMEOUT_MILLIS` | 5000ms | WebClient → OpenAI API 的 TCP 连接超时 |
| 读取超时 | `ReadTimeoutHandler` | 120s | 等待 OpenAI API 响应数据的最大时间 |
| 写入超时 | `WriteTimeoutHandler` | 30s | 向 OpenAI API 发送请求的最大时间 |
| 流超时 | `Flux.timeout()` | 可配置 (默认 120s) | 整体 SSE 流的最大持续时间 |
| 网关响应超时 | `gateway metadata.response-timeout` | 300000ms | Spring Cloud Gateway 等待上游响应的最大时间 |
| Nginx 代理超时 | `proxy_read_timeout` | 300s | Nginx 等待上游响应的最大时间 |

```java
// 文件: ai-assistant-service/src/main/java/com/kb/ai/config/WebClientConfig.java

@Bean
public HttpClient httpClient(ConnectionProvider connectionProvider) {
    return HttpClient.create(connectionProvider)
            .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 5000)
            .doOnConnected(conn ->
                    conn.addHandlerLast(new ReadTimeoutHandler(120, TimeUnit.SECONDS))
                            .addHandlerLast(new WriteTimeoutHandler(30, TimeUnit.SECONDS)));
}
```

### 2.6 客户端断开检测

当用户在前端点击"停止生成"或关闭标签页，浏览器会断开 EventSource 连接。Spring WebFlux 通过 `doOnCancel` 回调感知此事件：

```java
return aiStreamService.continueStream(request.getContext(), userId)
        .doOnCancel(() -> log.info("客户端断开连接: userId={}", userId));
```

在 `OpenAiStreamServiceImpl` 内部，`doOnCancel` 也会取消对 OpenAI API 的上游请求：

```java
return webClient.post()
        .uri("/v1/chat/completions")
        // ...
        .doOnCancel(() -> log.debug("上游 LLM 请求取消"));
```

Reactor Netty 的 `doOnCancel` 会触发底层 HTTP 连接的关闭信号，释放连接池资源。

### 2.7 熔断降级

使用 Resilience4j ReactiveCircuitBreaker 保护 OpenAI API 调用：

```java
// 文件: OpenAiStreamServiceImpl.java

// 滑动窗口 10 次调用，失败率 >= 50% → 熔断 30 秒
// 半开状态允许 3 次探测调用

return circuitBreaker.run(
        callOpenAiStream(context),
        throwable -> {
            log.warn("熔断/异常降级: {}", throwable.getMessage());
            return fallbackStream(throwable);  // 返回单条 SSE "AI服务暂不可用"
        }
);
```

降级策略返回一条友好的 SSE 消息，而非让客户端看到连接错误。

### 2.8 Demo 模式

当未配置 `OPENAI_API_KEY` 环境变量或 `AI_DEMO_MODE=true` 时，系统自动切换到 Demo 模式，返回预设的演示文本，以 200ms 间隔逐块推送，模拟真实 AI 续写的打字效果。

```java
// 文件: OpenAiStreamServiceImpl.java

private Flux<ServerSentEvent<String>> demoStream() {
    // 将预设文本切分为 2-4 字符的片段
    return Flux.fromStream(chunkText(text, 2, 4).stream())
            .delayElements(Duration.ofMillis(200))   // 模拟打字效果
            .map(chunk -> ServerSentEvent.<String>builder().data(chunk).build())
            .concatWith(Mono.just(
                    ServerSentEvent.<String>builder().data("[DONE]").build()));
}
```

此设计使得前端开发和演示无需真实 API Key。

### 2.9 连接池管理

```java
// 文件: ai-assistant-service/src/main/java/com/kb/ai/config/WebClientConfig.java

@Bean
public ConnectionProvider connectionProvider() {
    return ConnectionProvider.builder("ai-assistant-pool")
            .maxConnections(100)                           // 限制对 LLM 后端的并发连接
            .pendingAcquireTimeout(Duration.ofSeconds(10)) // 连接池耗尽时的等待超时
            .maxIdleTime(Duration.ofSeconds(60))           // 空闲连接回收

}
```

100 个连接的池上限基于 OpenAI API 的并发限制和本服务预期 QPS（估约 20-50 并发流）。

## 3. Redis Pub/Sub 跨实例缓存失效

### 3.1 问题背景

article-service 可水平扩展（docker-compose 中虽只有 1 个实例，但 `deploy: replicas: 3` 即可多实例）。当 interact-service 处理点赞请求，更新 Redis ZSET 中的热度分后，所有 article-service 实例的本地 Caffeine 缓存都需失效，确保下次查询 Top 10 时从 Redis 重新加载。

### 3.2 架构全貌

```
┌──────────────────────┐
│   Browser            │
│   POST /api/articles │
│       /1/like        │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│   Gateway            │
│   → interact-service │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────────────────────────────────────┐
│   interact-service: LikeServiceImpl.likeArticle()     │
│                                                      │
│   1. Redis SETNX (去重)                                │
│   2. Redis XADD (异步持久化)                            │
│   3. Redis ZINCRBY hot_articles <articleId> +3        │
│   4. Redis PUBLISH hot_articles:refresh "like:<id>"   │
└──────────┬───────────────────────────────────────────┘
           │
           │  Redis Pub/Sub
           ▼
┌──────────────────────────────────────────────────────┐
│   Redis Server                                        │
│   channel: hot_articles:refresh                       │
│                                                      │
│   消息广播至所有订阅者 (fan-out)                        │
└──────┬───────────────────────┬───────────────────────┘
       │                       │
       ▼                       ▼
┌─────────────────┐   ┌─────────────────┐
│ article-service │   │ article-service │
│   实例 #1       │   │   实例 #2       │
│                 │   │                 │
│ HotArticlesRe-  │   │ HotArticlesRe-  │
│ freshListener   │   │ freshListener   │
│   ↓             │   │   ↓             │
│ invalidate()    │   │ invalidate()    │
│ Caffeine cache  │   │ Caffeine cache  │
└─────────────────┘   └─────────────────┘
```

### 3.3 消息生产者

热度变更的三种触发来源：

**来源一：点赞** — `LikeServiceImpl.java`

```java

// 3. Update hot articles ZSET (+3 for a like)
redisTemplate.opsForZSet().incrementScore(

```

### 3.4 消息消费者

**频道订阅配置** — `RedisPubSubConfig.java`

```java
// 文件: article-service/src/main/java/com/kb/article/config/RedisPubSubConfig.java

@Bean
public RedisMessageListenerContainer redisMessageListenerContainer(
        RedisConnectionFactory connectionFactory) {
    RedisMessageListenerContainer container = new RedisMessageListenerContainer();
    container.setConnectionFactory(connectionFactory);
    container.addMessageListener(refreshListener,
            new ChannelTopic(RedisKeyConstants.HOT_ARTICLES_REFRESH_CHANNEL));
    log.info("Redis Pub/Sub listener registered on channel '{}'",
            RedisKeyConstants.HOT_ARTICLES_REFRESH_CHANNEL);
    return container;

```

`RedisMessageListenerContainer` 在独立线程中运行，不阻塞主业务线程。

**消息处理** — `HotArticlesRefreshListener.java`

```java
// 文件: article-service/src/main/java/com/kb/article/listener/HotArticlesRefreshListener.java

@Component
public class HotArticlesRefreshListener implements MessageListener {

    private final HotArticlesCacheManager cacheManager;

    @Override
    public void onMessage(Message message, byte[] pattern) {
        String body = new String(message.getBody(), StandardCharsets.UTF_8);
        String channel = new String(message.getChannel(), StandardCharsets.UTF_8);
        log.debug("Received refresh message on channel '{}': {}", channel, body);
        cacheManager.invalidate();   // 失效本地 Caffeine 缓存
    }
}
```

消息体（如 `"like:42"`、`"read:17"`、`"decay"`）当前仅用于日志追踪，核心动作是统一的 `invalidate()`。

### 3.5 Caffeine 缓存管理器

```java
// 文件: article-service/src/main/java/com/kb/article/cache/HotArticlesCacheManager.java

@Component
public class HotArticlesCacheManager {

    private final int cacheTtlSeconds;   // 默认 1 秒，可配置
    private Cache<String, List<TopArticleVO>> cache;

    @PostConstruct
    public void init() {
        this.cache = Caffeine.newBuilder()
                .expireAfterWrite(Duration.ofSeconds(cacheTtlSeconds))
                .maximumSize(1)       // 只缓存一个 key: "top10"
                .recordStats()
                .build();
    }

    public List<TopArticleVO> getOrCompute(Supplier<List<TopArticleVO>> supplier) {
        return cache.get("top10", key -> {
            log.debug("Cache miss for top10, computing...");
            return supplier.get();   // → fetchTop10FromRedis()
        });
    }

    public void invalidate() {
        cache.invalidateAll();
    }
}
```

### 3.6 传播延迟分析

| 环节 | 延迟 |
|------|------|
| Redis PUBLISH 命令 | < 1ms |
| Redis 消息分发至订阅者 | < 1ms (单机 Redis，同网络) |
| `onMessage` → `invalidateAll()` | < 0.1ms |
| 合计 | < 2ms (亚毫秒级) |

这保证了"一次点赞 → 所有实例缓存立即失效"的实时性。

### 3.7 最终一致性保证 — Caffeine TTL 兜底

**问题**：如果 Redis Pub/Sub 消息丢失（Redis 重启、网络瞬断、listener 临时不可用），缓存将不会主动失效，导致多实例间的数据不一致。

**解决方案**：Caffeine 缓存 TTL 设置为 1 秒。即使 Pub/Sub 消息丢失，最坏情况下缓存会在 1 秒后自动过期，下次查询从 Redis ZSET 重新加载。这是一个"最终一致性"保证：

> **at-most-1-second staleness**：任何时刻，缓存中的数据与实际 Redis ZSET 中数据的偏差不超过 1 秒。

这个设计的核心理念：
- Pub/Sub 提供"即时一致性"（best-effort，< 2ms）
- Caffeine TTL 提供"最终一致性"（guaranteed，<= 1s）
- 两层机制互补，既有实时性又有可靠性

## 4. 设计决策与权衡

### 4.1 为什么用 SSE 而非 WebSocket？

| 维度 | SSE (选择) | WebSocket |
|------|-----------|-----------|
| 通信方向 | 单向（服务器→客户端），匹配续写场景 | 双向，但客户端无需向服务器推送 |
| 协议复杂度 | HTTP 长连接，浏览器原生 EventSource API | 需要升级协议，需额外库 |
| 代理兼容性 | 良好，Nginx 只需关闭缓冲 | 需要代理支持 WebSocket 升级 |
| 自动重连 | EventSource 内置自动重连 | 需手动实现 |
| 负载均衡 | 更友好（纯 HTTP） | 需要 sticky session |

AI 续写是纯粹的单向数据推送场景，SSE 是更自然的选择。

### 4.2 为什么用 Redis Pub/Sub 而非消息队列？

| 维度 | Redis Pub/Sub (选择) | RabbitMQ |
|------|---------------------|----------|
| 消息语义 | 即发即忘 (fire-and-forget) | 持久化 + 确认 |
| 是否需要持久化 | 否 — 丢失可用 TTL 兜底 | 是 — 但这里不需要 |
| 部署复杂度 | 已有 Redis，零额外成本 | 需额外配置 exchange/queue/binding |
| 延迟 | < 1ms | 1-10ms |

缓存失效消息不需要持久化（丢失有 TTL 兜底），Redis Pub/Sub 是更轻量的选择。

### 4.3 什么时候需要升级？

- **SSE → WebSocket**：当需要双向实时通信时（如协同编辑、实时评论通知），切换到 WebSocket。
- **Pub/Sub → 消息队列**：当缓存失效需要可靠投递、不能容忍 TTL 窗口时（如金融数据缓存），切换到 RabbitMQ 持久化队列。
